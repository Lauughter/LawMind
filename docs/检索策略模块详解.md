# LawMind 检索策略模块详解

## 一、总体架构

检索模块采用**多级流水线 + 多路召回 + 融合排序**的架构，整个流程分为 6 个阶段：

```
用户问题 → 预处理 → 热点缓存 → 向量化 → 多路召回 → 融合排序 → LLM生成
```

核心编排器位于 `RagServiceImpl.java`，两条路径：**同步路径** (`processQuestion()`, 行244) 和 **SSE 流式路径** (`processQuestionStream()`, 行857)。

---

## 二、查询预处理阶段

### 2.1 文本清洗（`sanitizeUserInput()`）
- 移除 Markdown 代码块标记
- 过滤注入攻击模式（忽略/覆盖系统提示词等）
- 长度截断至 **2000 字符**

### 2.2 法律相关性判定（`isLegalRelatedQuestion()`）
- 约 **150 个法律关键词**的纯规则匹配
- 非法律问题直接返回固定拒绝回复，来源标记 `non_legal_reject`

### 2.3 结构化预处理（`TextPreprocessUtil.preprocess()`）

| 步骤 | 操作 | 示例 |
|------|------|------|
| 去语气词 | 移除"请问""我想知道""谢谢"等14个模式 | — |
| 术语标准化 | 口语→正式法律术语 | "五险一金"→"社会保险和住房公积金" |
| 特殊字符清理 | 正则 `[^一-龥a-zA-Z0-9\s]` | — |
| 英文小写化 | 统一转小写 | — |

同时生成预处理后文本的 **MD5** 哈希，作为热点缓存键。

### 2.4 意图分类（`IntentClassifier`）
纯关键词匹配，识别 6 种意图：

| 意图 | 对检索的影响 |
|------|-------------|
| `LEGAL_CONSULTATION` | 默认 topK=10 |
| `ARTICLE_LOOKUP` | topK **+5**（最少15条） |
| `CASE_SEARCH` | topK **+3**（最少13条） |
| `CALCULATION` | topK **-2**（最少3条） |
| `DOCUMENT_GENERATION` | 默认 |
| `CONTRACT_REVIEW_RELATED` | 默认 |

### 2.5 实体抽取（`LegalEntityExtractor`）
从查询中抽取结构化信息，用于后续检索过滤：
- **lawType**：120+关键词映射到完整法律名（如"劳动法"→"中华人民共和国劳动法"）
- **articleReference**：正则提取法条引用（`《XX法》第X条`）
- **amountText**：金额相关表述
- **partyType**：当事人类型（用人单位/劳动者/消费者等）

### 2.6 查询扩展（`LegalQueryExpander.expandQuery()`）
- 57 组口语→正式术语映射，**追加式扩展**（不替换原问题）
- 例如："被公司开除了怎么办" → 追加 "解除劳动合同 辞退 用人单位单方解除"

## 三、热点缓存层（第一级召回）

```
Redis Key: hot:question:{md5}
TTL: 30天（rag.hot-cache.initial-ttl-days）
```

- 用预处理后的 MD5 直接查 Redis
- **命中即直接返回**，跳过后续所有检索步骤
- 触发升级条件（`VisitStatsUtil.isHotQuestion()`）：

| 时间窗口 | 阈值（默认） |
|----------|------------|
| 5 分钟内 | 3 次 |
| 1 小时内 | 10 次 |
| 1 天内 | 30 次 |

## 四、查询改写与向量化

### 4.1 LLM 查询改写（`rewriteQueryWithLLM()`）
- Prompt 来自 `sys_config` 表 `rag.query_rewrite_prompt` 或内置默认值
- 将口语化问题改写为适合检索的形式化表述
- 失败时**回退到 `LegalQueryExpander` 规则扩展**

### 4.2 向量化（`EmbeddingUtil.embed()`）
- 模型：阿里云 DashScope **text-embedding-v2**
- 输出：**1536 维** FLOAT32 向量
- 通过 LangChain4j Spring Boot Starter 自动配置

### 4.3 HyDE 增强（可选，默认关闭）

```
if hydeEnabled:
    LLM 生成假设法律分析(200-400字) → embed → knowledgeVector
else:
    knowledgeVector = questionVector
```

- 知识库检索使用 `knowledgeVector`，相似问题检索始终使用 `questionVector`
- 配置项：`rag.search.hyde.enabled: false`

## 五、多路召回（第二级）

### 5.1 相似问题检索

```
索引: idx:similar_question (Redis Stack)
前缀: similar:question:
距离: COSINE
匹配阈值: 0.85（rag.threshold.similar-question）
```

- 使用**问题向量**查询 Redis KNN
- 额外检查：问题类型匹配（`classifyQuestionType()`）
- 平局时随机挑选

### 5.2 混合搜索（核心召回）

`HybridSearchService.searchHybridFiltered()` 融合两路：

#### 路 1：Redis 向量搜索（语义匹配）

```
索引: idx:law_knowledge
前缀: law:vector:
算法: FLAT KNN, COSINE 距离
FetchSize: topK*4（有 lawType 过滤时）或 topK*2（无过滤时）
```

- 距离→相似度转换：`similarity = max(0, min(1, 1 - distance/2))`
- 结果检查 `status=EFFECTIVE` 和 lawType 匹配

#### 路 2：MySQL 全文搜索（关键词匹配）

```sql
MATCH(title, content) AGAINST('+term1 +term2 ...' IN BOOLEAN MODE)
```

- 使用 MySQL **ngram** 分词器（适配中文）
- 查询按空格和中文标点切分为语义单元
- 保留法律引用模式（如"第XXX条"）作为原子单元
- 所有词必须 ≥ 2 字符（`NGRAM_MIN_LENGTH`），否则跳过全文搜索

#### RRF 融合公式（k=60）

```
RRF_score = 1/(60 + rank_vector) + 1/(60 + rank_fulltext)
```

然后通过 **Min-Max 归一化**映射到 0-1 范围，兼容后续阈值配置。

## 六、融合排序（第三级）

### 6.1 MMR 多样化（`SearchResultDiversifier`）

```
MMR = λ × relevance - (1-λ) × diversityPenalty
λ = 0.7（偏向相关性）
```

文档相似度计算（`computeDocumentSimilarity`）：

| 条件 | 相似度 |
|------|-------|
| 相同 lawType + 相同 chapter | 1.0 |
| 相同 lawType + 不同 chapter | 0.5 |
| 不同 lawType | 0.0 |

采用**贪心迭代**算法逐个选入已选集。

### 6.2 双阈值过滤（`filterByThreshold()`）

```
高质量阈值: 0.70（rag.threshold.law-knowledge）
   ↓ ≥ 0.70 → 直接采用

绝对最低阈值: 0.65（rag.threshold.filter）
   ↓ ≥ 0.65 且 < 0.70 → 边缘结果（高质量为空时兜底）

   < 0.65 → 丢弃
```

## 七、LLM 生成与后处理

### 7.1 生成路径
- **有知识结果**：`generateAnswerWithTokens()` — 携带系统提示 + 检索到的知识 + 用户问题
- **无知识结果**：`generateDirectAnswerWithTokens()` — 仅系统提示 + 用户问题，标记来源 `llm_direct`
- **流式路径**：使用 `StreamingChatLanguageModel`（同为 qwen-plus），通过 SseEmitter 推送，事件类型：`token` / `knowledge` / `done` / `error`

### 7.2 引用验证（`verifyCitations()`）
- 正则提取：`(?:《([^》]{1,30})》)?\s*第([一二三四五六七八九十百千万零〇\d]+)条`
- 与检索到的 `LawKnowledge` 交叉比对（法律名 + 条文号）
- 中文数字→阿拉伯数字转换（`chineseNumToArabic()`）
- 未匹配的引用追加 **"⚠ 注意：以下法律引用未经检索结果验证"** 警告

### 7.3 合规声明
- `law_knowledge` 和 `llm_direct` 来源的回答自动附加免责声明

```yaml
# rag 检索配置
rag:
  vector:
    dimension: 1536
  search:
    top-k: 10
    hybrid:
      enabled: true
    mmr:
      enabled: true
      lambda: 0.7
    hyde:
      enabled: false
  threshold:
    similar-question: 0.85    # 相似问题最低匹配度

    filter: 0.65              # 绝对最低阈值

    enabled: true

      enabled: true

    1-hour: 10
    1-day: 30
  hot-cache:
    initial-ttl-days: 30
  conversation:
    max-history-messages: 10  # 多轮对话上下文
```

## 十、关键设计决策

1. **为什么用 RRF 而不是加权求和？** 向量分和全文分的数值分布差异大，直接加权需要校准。RRF 是 rank-based，天然免疫分数尺度差异，且不需要额外训练。

2. **为什么混合搜索的 fetchSize 要放大 2-4 倍？** RRF 融合 + MMR + 双阈值过滤会层层淘汰结果。取更多候选（40条向量 + MySQL全文）保证经过过滤后仍有足够的优质结果。

3. **为什么 HyDE 默认关闭？** HyDE 需要额外一次 LLM 调用（假设文档生成），增加 1-3 秒延迟和 token 消耗。只有在原始查询过于简短/口语化时才值得开启。

4. **双阈值设计的意图？** 高质量阈值保证回答基于立场明确的知识；边缘结果在无更好选择时作为兜底，避免因阈值过高导致"无结果"。

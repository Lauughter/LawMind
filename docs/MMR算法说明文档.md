# MMR（最大边际相关性）算法说明

## 一句话解释

MMR 在挑选搜索结果时，既考虑"跟查询相不相关"，也考虑"跟已经选中的结果重不重复"。它会在相关性和多样性之间取一个平衡，避免返回一堆高度相似的冗余结果。

---

## 要解决什么问题？

假设用户问："**被开除后怎么维权？**"，混合搜索返回 Top-5 结果：

| 排名 | 法律 | 章节 | 相关性分数 |
|------|------|------|-----------|
| 1 | 劳动合同法 | 第四章 劳动合同的解除 | 0.92 |
| 2 | 劳动合同法 | 第四章 劳动合同的解除 | 0.89 |
| 3 | 劳动合同法 | 第四章 劳动合同的解除 | 0.85 |
| 4 | 劳动合同法 | 第七章 法律责任 | 0.82 |
| 5 | 劳动争议调解仲裁法 | 第二章 调解 | 0.78 |

**问题：** 前 3 条都是同一部法律同一章节，内容高度重复。用户想知道的信息（经济补偿标准、仲裁流程、维权步骤）可能被挤到了第 4、5 名甚至更后面。

**MMR 的目标：** 重排后的结果更分散，例如——

| 排名 | 法律 | 章节 |
|------|------|------|
| 1 | 劳动合同法 | 第四章 劳动合同的解除 |
| 2 | 劳动争议调解仲裁法 | 第二章 调解 |
| 3 | 劳动合同法 | 第七章 法律责任 |
| 4 | 社会保险法 | 第五章 失业保险 |
| 5 | 劳动合同法实施条例 | — |

每一条来自不同的法律或者不同的章节，信息覆盖面大幅提升。

---

## 公式

```
MMR = λ × 相关性分数  −  (1−λ) × 与已选文档的最大相似度
```

**符号说明：**

| 符号 | 含义 |
|------|------|
| `doc` | 当前正在评估的候选文档 |
| `relevance(doc)` | 文档与用户查询的相关性分数（混合搜索给的分数，已归一化到 0-1） |
| `similarity(doc, selected)` | 文档与**某个已选中文档**之间的相似度 |
| `max(...)` | 取候选文档与**所有已选中文档**中相似度的**最大值**（最相似的那个） |
| `λ` (lambda) | 权衡参数，范围 0-1 |

**公式解读（分两步看）：**

- 前半部分 `λ × relevance(doc)` → **奖励相关性**：和查询越相关，MMR 分数越高
- 后半部分 `(1-λ) × max_sim` → **惩罚重复**：和已选中文档越像，MMR 扣分越多
- 最终取 **MMR 分数最高** 的候选文档加入结果集

**λ 取值的含义：**

| λ | 效果 |
|---|------|
| 1.0 | 只看相关性，等价于原始排序（MMR 不生效） |
| 0.7 | **推荐值**：70% 相关性 + 30% 多样性（当前配置） |
| 0.5 | 均衡：相关性和多样性各占一半 |
| 0.0 | 只看多样性，完全不考虑相关性（极端，不推荐） |

## 逐步演示

用上面的例子，假设 λ=0.7，5 个候选文档（A 到 E，相关性分数已归一化）：

```
候选列表: [A(0.92, 劳动合同法, 第四章), B(0.89, 劳动合同法, 第四章),
          C(0.85, 劳动合同法, 第四章), D(0.82, 劳动合同法, 第七章),
          E(0.78, 劳动争议调解仲裁法, 第二章)]
```

**文档相似度规则（`computeDocumentSimilarity`）**：

```
相同法律类型 + 相同章节 → 1.0 （高度重复）
相同法律类型 + 不同章节 → 0.5 （部分重复）
不同法律类型           → 0.0 （不重复）
```

### 第 1 轮（选第 1 个）

此时没有已选中的文档，所以 `diversityPenalty = 0`，MMR 公式退化为 `λ × relevance`。所有文档按相关性排序，选相关性最高的 **A**。

### 第 2 轮（选第 2 个）

已选中: `[A]`
候选: `[B, C, D, E]`

| 候选 | 相关性 | 与A的相似度 | MMR 计算 |
|------|--------|-------------|----------|
| B | 0.89 | 1.0（同法同章） | 0.7 × 0.89 − 0.3 × **1.0** = **0.323** |
| C | 0.85 | 1.0（同法同章） | 0.7 × 0.85 − 0.3 × **1.0** = **0.295** |
| D | 0.82 | 0.5（同法不同章） | 0.7 × 0.82 − 0.3 × **0.5** = **0.424** |
| E | 0.78 | 0.0（不同法） | 0.7 × 0.78 − 0.3 × **0.0** = **0.546** |

> 选 **E**（MMR=0.546）

### 第 3 轮（选第 3 个）

已选中: `[A, E]`
候选: `[B, C, D]`

对于每个候选，取它与 A 和 E 相似度的**最大值**：

| 候选 | 相关性 | 与A相似度 | 与E相似度 | max_sim | MMR 计算 |
|------|--------|-----------|-----------|---------|----------|
| B | 0.89 | 1.0 | 0.0 | 1.0 | 0.7 × 0.89 − 0.3 × 1.0 = **0.323** |
| C | 0.85 | 1.0 | 0.0 | 1.0 | 0.7 × 0.85 − 0.3 × 1.0 = **0.295** |
| D | 0.82 | 0.5 | 0.0 | 0.5 | 0.7 × 0.82 − 0.3 × 0.5 = **0.424** |

> 选 **D**（MMR=0.424）

### 第 4 轮

已选中: `[A, E, D]`
候选: `[B, C]`

| 候选 | max_sim | MMR |
|------|---------|-----|
| B | 1.0（与A） | 0.323 |
| C | 1.0（与A） | 0.295 |

> 选 **B**

### 最终结果

原始排序 → MMR 重排后：

```
原始: A(0.92) → B(0.89) → C(0.85) → D(0.82) → E(0.78)
      全部挤在"劳动合同法第四章"
MMR:  A(0.92) → E(0.78) → D(0.82) → B(0.89) → C(0.85)
      跨了 3 部法律 / 2 个章节，信息覆盖面大得多
```

## 与代码的对应关系

### `diversify()` 方法

```java
public List<LawKnowledge> diversify(
        List<LawKnowledge> results,   // 混合搜索返回的原始排序结果
        String queryText,             // 用户查询（预留，当前未使用）
        int topK,                     // 最终返回多少条
        double lambda) {              // λ 权衡参数
```

**执行流程：**

```
1. 归一化所有相关性分数到 [0, 1]
2. 创建 remaining（候选池）= 所有结果
3. 创建 selected（已选池）= 空
4. 循环 topK 轮或候选池耗尽:
   a. 对候选池中每个文档，计算 MMR 分数
   b. 选 MMR 最高的文档，移入已选池
   c. 记录日志
5. 返回已选池
```

**时间复杂度：** O(K × N²)，N 是候选数，K 是 topK。在这个场景下 N ≤ 20，K ≤ 10，完全可接受。

### `computeDocumentSimilarity()` 方法

```java
private double computeDocumentSimilarity(LawKnowledge a, LawKnowledge b) {
    // 不同法律类型 → 0.0
    if (!lawTypeA.equals(lawTypeB)) return 0.0;

    // 相同法律 + 相同章节 → 1.0
    if (chapterA != null && chapterA.equals(chapterB)) return 1.0;
    if (chapterA == null && chapterB == null) return 1.0;  // 都没有章节信息

    // 相同法律 + 不同章节 → 0.5
    return 0.5;
}
```

这是一个**离散的三值相似度**（0.0 / 0.5 / 1.0），不是真正的语义相似度。选择这个方案的原因：

- **简洁高效**：无需再次调用 embedding 模型，计算量为 O(1)
- **适合法律场景**：法律文本的结构化特征（法律类型、章节）本身就是很好的多样性维度
- **可扩展**：未来可以将 `lawType` 和 `chapter` 替换为实际的文本语义相似度

## 代码中的调用位置

在 `RagServiceImpl.searchLawKnowledge()` 中：

```java
public List<LawKnowledge> searchLawKnowledge(float[] questionVector, String expandedQuery) {
    // 1. 混合搜索（向量 + 全文 RRF 融合）
    rawResults = hybridSearchService.searchHybrid(questionVector, expandedQuery, topK);

    // 2. MMR 多样化（结果数 > 1 时生效）
    if (ragConfig.isMmrEnabled() && rawResults.size() > 1) {
        rawResults = searchResultDiversifier.diversify(
                rawResults, expandedQuery, topK, ragConfig.getMmrLambda());
    }

    // 3. 双阈值过滤（0.70 / 0.65）
    return filterByThreshold(rawResults);
}
```

**关键设计：** MMR 在双阈值过滤**之前**执行。这样 MMR 可以看到完整的候选集，避免"某个法律章节的高分结果全部被阈值过滤掉后，其他法律的结果连进入候选池的机会都没有"。

## 配置调整指南

在 `application.yml` 中调整：

```yaml
rag:
  search:
    mmr:
      enabled: true       # false = 关闭 MMR，直接用原始排序
      lambda: 0.7         # 调整相关性与多样性的平衡
```

**调参建议：**

| 场景 | 建议 λ | 理由 |
|------|--------|------|
| 法律条文库比较单一（只有几部法律） | 0.6 | 更需要多样性避免重复 |
| 法律条文库丰富（多领域法律） | 0.8 | 相关性更重要 |
| 用户问的是具体法条查询（"劳动合同法第39条"） | 1.0（关掉） | 精确查询不需要多样性 |
| 用户问的是开放性法律问题 | 0.6-0.7 | 需要多角度看问题 |

## 总结

| 对比维度 | 无 MMR | 有 MMR |
|----------|--------|--------|
| 结果集中度 | 可能全来自同一法律同一章节 | 分散到不同法律和章节 |
| 信息覆盖率 | 低（冗余信息多） | 高（角度多样） |
| LLM 回答质量 | 知识面窄，容易片面 | 知识面广，回答更全面 |
| 额外计算量 | 0 | O(K × N²)，K≤10, N≤20，可忽略 |
| 可控性 | 无 | 通过 λ 参数灵活调节 |

# LawMind 质量评估体系优化方案

> 参照《企业级RAG项目差距分析与优化路线图》§2.3，基于项目当前代码的深度审计结果制定。
>

---

## 一、当前状态诊断

### 1.1 差距总览

| 维度 | 路线图目标 | 当前状态 | 差距等级 |
|------|-----------|---------|---------|
| Golden Dataset | 100-200 条标注问答对，每次发版前回归 | **存在但未接入** — `docs/golden-dataset-rag-evaluation.json` 有 40 条标注数据，但零代码消费 | **严重** |
| RAGAS 评估 | 自动化计算 Faithfulness、Context Precision/Recall | **不存在** — 无任何代码引用 RAGAS | **严重** |
| 反馈闭环 | 用户点踩 → 审核 → 修正知识库 / 加入回归集 | **管道断裂** — 点踩入库后无自动动作，审核靠人工手动查 | **严重** |
| A/B 测试 | 两套 Prompt/参数对比点赞率 | **不存在** — 无实验框架、无特征开关、无分桶路由 | 缺失 |
| 在线指标看板 | 缓存命中率、来源占比、点赞率趋势 | **仅运维指标** — 有延迟/来源/反馈计数，但无回答质量指标 | **严重** |
| 自动化测试 | CI 集成、每次改动跑评估 | **不存在** — 无 CI 配置，单元测试仅 smoke 级别 | **严重** |

### 1.2 逐项详细诊断

#### 1.2.1 反馈系统 — 数据流在入库后断裂

```
当前数据流：
  用户点踩 → POST /ai-chat/feedback → MySQL feedback=-1, feedback_status='PENDING_REVIEW'
       ↓
  [数据流终止] ← 这里是断裂点
       ↓
  管理员手动访问 /admin/review-queue → 人工查看 → 标记 REVIEWED/DISMISSED
       ↓
  [数据流再次终止] ← 审核结果没有任何后续动作
```

**具体问题：**

| # | 问题 | 影响 |
|---|------|------|
| 1 | 前端未采集 `feedbackContent` | `MessageActions.vue` 只有点赞/点踩按钮，无文字输入框。即使用户想解释为什么点踩，也无法输入。后端 `AiChatController` 接受 `feedbackContent` 参数但前端从未发送。 |
| 2 | 点踩后无通知机制 | 管理员不会收到任何通知（无邮件、无站内信、无 Webhook），只能主动去查 `/admin/review-queue`。 |
| 3 | 审核结果无后续动作 | `submitReview()` 只写入 `feedback_status` 和 `review_notes`，不触发知识库修正、不更新 Golden Dataset、不调整检索参数。 |
| 4 | 点赞数据也未利用 | `feedback=1` 的记录仅统计入 Redis，未用于识别高质量回答以反哺 Few-shot 示例库。 |
| 5 | 无反馈聚合分析 | 无法回答"哪种类型的问题点踩率最高""哪个知识来源的点踩率最低"等管理问题。 |

#### 1.2.2 Golden Dataset — 数据已存在但未接入

**关键发现：** `docs/golden-dataset-rag-evaluation.json` 包含 **40 条**精心标注的测试用例，数据结构如下：

```json
{
  "id": "gd-001",
  "question": "公司拖欠我三个月工资，我该怎么办？",
  "intent": "LEGAL_CONSULTATION",
  "expected_law_type": "劳动法",
  "difficulty": "easy",
  "key_points": ["劳动报酬", "拖欠工资", "劳动仲裁"],
  "expected_answer_contains": ["劳动仲裁", "工资支付", "解除劳动合同"],
  "source_requirement": "law_knowledge"
}
```

**问题：零代码消费。** 没有任何 Java 测试、Python 脚本、或 CI 流程读取这个文件。它是一个"设计文档"，不是"测试资产"。

**缺失项：**
- 没有评估执行器（读取 Golden Dataset → 调用 RAG 管道 → 对比期望输出）
- 没有评分逻辑（如何给回答打分？按 key_points 覆盖率？按 expected_answer_contains 命中率？）
- 没有回归测试集成（改一行检索代码后无法自动验证是否退化）

#### 1.2.3 RAGAS 评估 — 完全不存在

**当前项目依赖中无 RAGAS。** `pom.xml` 中没有 `ragas` 或任何评估框架的依赖。整个项目中 RAGAS 只出现在路线图文档的规划描述中。

RAGAS 需要的四类数据，当前项目**全不满足**：

| RAGAS 所需输入 | 当前项目是否可提供 |
|---------------|-------------------|
| `question` | 可提供（用户问题原文） |
| `answer` | 可提供（LLM 生成的回答） |
| `contexts`（检索到的文档列表） | **不可直接提供** — 检索结果未持久化存储，仅在请求生命周期内存存在 |
| `ground_truth`（标准答案） | **不可提供** — Golden Dataset 未被接入 |

#### 1.2.4 单元测试 — 仅烟雾测试级别

`RagServiceImplTest.java` 共 4 个测试，全部是结构断言：

| 测试 | 断言类型 | 是否验证质量 |
|------|---------|-------------|
| `nonLegalQuestion_rejected` | 字符串包含 | 否 |
| `hotCacheHit_returnsCachedAnswer` | 值相等 | 否 |
| `fullPipeline_addsDisclaimer` | 字符串包含 | 否 |
| `knowledgeFound_feedsIntoAnswer` | 非空判断 | 否 |

没有任何测试验证：回答的事实正确性、引用的法条是否准确、检索结果的相关性。

#### 1.2.5 在线指标 — 运维指标齐全，质量指标为零

`RagMetricsServiceImpl` 收集的指标全部属于**运维维度**：

```
已有（运维指标）：              缺失（质量指标）：
├── 总请求数                     ├── 回答准确率
├── 来源分布（缓存/检索/LLM）      ├── 幻觉率
├── 平均延迟 / P50 / P95          ├── 引用准确率
├── 延迟分阶段（预处理/嵌入/检索/生成）├── 上下文精确率
├── 检索结果数 / 最高相似度        ├── 上下文召回率
├── HyDE 使用次数                 ├── 回答相关性得分
├── 点赞/点踩计数                 ├── 复问率（同一用户追问）
└── 点赞率                       └── 点踩后退 rate
```

#### 1.2.6 CI/CD — 完全不存在

- 无 `.github/workflows` 目录
- 无 `Jenkinsfile`
- 无 Docker Compose 测试编排
- `mvn test` 的结果无任何自动化消费

## 二、优化方案

### Phase A：Golden Dataset 接入（优先级 P0，预计 2 天）

#### A1. 创建评估执行器

**目标：** 让已有的 40 条 Golden Dataset 能够自动化执行。

**方案：** 在 `src/test/java` 下新建 `evaluation/GoldenDatasetEvaluator.java`，基于 JUnit 5 `@ParameterizedTest`。

```
输入: docs/golden-dataset-rag-evaluation.json
流程:
  1. 逐条读取测试用例
  2. 调用 RagService.processQuestion(question) 获取完整回答
  3. 对每个测试用例执行多维度评分
  4. 汇总报告

评分维度（针对每条用例）:
  ├── source_match:     实际来源是否符合 source_requirement（0或1，硬性指标）
  ├── keyword_recall:   expected_answer_contains 中的关键词在回答中的命中率（0.0-1.0）
  ├── law_type_match:   expected_law_type 是否出现在检索到的知识中（0或1）
  └── answer_length:    回答长度是否 ≥ 50 字符（0或1，防止空回答）

汇总报告:
  ├── 总用例数 / 通过数 / 失败数
  ├── 每个维度的平均分
  ├── 每条失败用例的详细信息（期望 vs 实际）
  └── JSON 格式的详细报告输出到 target/golden-eval-report.json
```

**关键设计决策：**

1. **不依赖外部 LLM 评分（RAGAS 风格）** — Phase A 用纯规则评分（关键词匹配 + 来源验证），不需要额外 LLM 调用，执行快速（40 条 < 3 分钟），适合作为 CI 门禁。
2. **使用 `@Tag("evaluation")` 标记** — 与快速单元测试分离，评估测试运行慢（涉及实际 LLM 调用），不适合每次 commit 都跑。
3. **依赖真实的 Spring 上下文**（`@SpringBootTest`）而非 Mock，因为需要真实的 Embedding、Redis、LLM。

#### A2. 扩展 Golden Dataset

当前 40 条，目标 100-200 条。扩展方向：

| 优先级 | 法律领域 | 建议新增条数 | 理由 |
|--------|---------|------------|------|
| 高 | 婚姻家庭（离婚、财产分割、抚养权） | 10 | 高频咨询领域 |
| 高 | 房产纠纷（买卖、租赁、物业） | 10 | 高频咨询领域 |
| 高 | 债权债务（借贷、担保、追偿） | 10 | 高频咨询领域 |
| 中 | 消费维权（退一赔三、产品质量） | 8 | 民生热点 |
| 中 | 行政执法（处罚、复议、诉讼） | 5 | 专业深度 |
| 低 | 知识产权（商标、专利、著作权） | 5 | 企业用户刚需 |
| 低 | 边界 case（超长问题、纯数字、emoji） | 12 | 鲁棒性测试 |

**新增字段建议：**

```json
{
  "expected_source": "law_knowledge",       // 更精确的来源期望
  "forbidden_content": ["刑法第XX条"],       // 不应出现的内容（负向测试）
  "min_retrieval_count": 2,                 // 至少应检索到的知识条数
  "tags": ["劳动法", "工资", "仲裁"]          // 便于分类统计
}
```

#### A3. 集成到开发流程

```
开发流程变更：
  feature 分支 → mvn test（快速单元测试，< 30s）→ 通过
       ↓
  PR 提交 → mvn test -P evaluation（评估测试，< 5min）→ 通过
       ↓
  合并到 main → 生成评估报告 → 与上次报告对比 → 无退化告警
```

`pom.xml` 中新增 Maven profile：

```xml
<profile>
  <id>evaluation</id>
  <build>
    <plugins>
      <plugin>
        <groupId>org.apache.maven.plugins</groupId>
        <artifactId>maven-surefire-plugin</artifactId>
        <configuration>
          <groups>evaluation</groups>
        </configuration>
      </plugin>
    </plugins>
  </build>
</profile>
```

### Phase B：RAGAS 评估集成（优先级 P1，预计 2 天）

#### B1. 技术选型

**当前约束：** LawMind 是 Java 17 + Spring Boot 项目，但 RAGAS 原生是 Python 库。

**方案对比：**

| 方案 | 优点 | 缺点 |
|------|------|------|
| **方案1：Python 子进程** — Java 调用 Python 脚本执行 RAGAS | RAGAS 完整功能，社区最佳实践 | 需要 Python 环境，跨语言调试复杂 |
| **方案2：Java 自行实现 RAGAS 逻辑** | 纯 Java，无额外依赖 | RAGAS 的核心是 LLM-as-judge，自行实现工作量约 3 天 |
| **方案3（推荐）：自建 Faithfulness + Relevance 评分** | 利用已有的 chatLanguageModel，在 Java 侧实现核心 RAGAS 指标 | 不需要 Python，指标计算方式可定制 |

**推荐方案 3** — 原因：
1. LawMind 已有 `chatLanguageModel`（qwen-plus），可直接用于 LLM-as-judge
2. RAGAS 最核心的两个指标（Faithfulness + Answer Relevance）可以用结构化 Prompt 在 Java 侧实现
3. Context Precision 和 Context Recall 需要人工标注的 ground_truth，当前 Golden Dataset 的 `key_points` 字段可替代
4. 避免引入 Python 依赖和跨语言调用复杂度

#### B2. 实现架构

```
新增类:
  evaluation/
  ├── RagasMetrics.java          # 评估指标数据类
  │   - faithfulness: double     # 忠实度 0-1
  │   - answerRelevance: double  # 回答相关性 0-1
  │   - contextPrecision: double # 上下文精确率 0-1（需要 ground_truth）
  │   - contextRecall: double    # 上下文召回率 0-1（需要 ground_truth）
  │
  ├── FaithfulnessEvaluator.java # 忠实度评估器
  │   - 输入: answer(List<String> statements), contexts
  │   - 方法: 对回答逐句判断是否可从 context 推断
  │   - 实现: LLM-as-judge → 结构化输出 → 计算比例
  │
  ├── AnswerRelevanceEvaluator.java # 回答相关性评估器
  │   - 输入: question, answer
  │   - 方法: 基于 answer 反向生成问题 → 计算与原问题的余弦相似度
  │   - 实现: LLM 生成反向问题 + EmbeddingUtil 计算相似度
  │
  └── EvaluationReport.java      # 评估报告生成器
      - 聚合多指标 → JSON/Markdown 报告
```

**Faithfulness Prompt 设计：**

```
System: 你是一个严格的法律事实核查员。我会给出一段 AI 回答和对应的法律知识原文。
请逐句判断回答中的每一句话是否能从法律知识原文中直接推断或证实。

回答：
{answer}

法律知识原文：
{contexts}

请用 JSON 格式输出，每条陈述包含 statement（原句）、verdict（"FAITHFUL"/"UNFAITHFUL"）、reason（判定理由）。
```

#### B3. 评估报告集成到 Golden Dataset 执行器

在 Phase A 的 `GoldenDatasetEvaluator` 中追加两个维度：

```
新增评分维度:
  ├── faithfulness:     FaithfulnessEvaluator 评分（需要 LLM 调用，较慢）
  ├── answer_relevance: AnswerRelevanceEvaluator 评分（需要 LLM + Embedding）
  └── 这两个维度只在 -P evaluation 时启用，不在快速单元测试时执行
```

#### B4. 评估数据持久化

```
新增表: evaluation_report
  id, run_id, created_at, golden_dataset_version,
  avg_faithfulness, avg_answer_relevance, avg_keyword_recall,
  avg_source_match, total_cases, passed_cases, failed_cases,
  report_json (完整评估报告 JSON)

用途:
  - 每次发版前跑一次评估 → 写入 evaluation_report
  - 可视化：画 Faithfulness 和 Relevance 的趋势图
  - 回退决策：如果 Faithfulness 从 0.85 跌到 0.70，回退本次修改
```

### Phase C：反馈闭环打通（优先级 P1，预计 1.5 天）

#### C1. 前端增强 — 采集点踩原因

`MessageActions.vue` 修改方案：

```
当前：
  点踩按钮 → 直接发送 { chatId, feedback: -1 }

修改后：
  点踩按钮 → 弹出简单的反馈面板
    ├── 预设选项（可多选）：
    │   ○ 回答不准确
    │   ○ 引用的法条有误
    │   ○ 没有回答我的问题
    │   ○ 回答太笼统
    │   ○ 其他
    └── 可选文本框（补充说明）
  → 发送 { chatId, feedback: -1, feedbackContent: "回答不准确,引用的法条有误|具体说明..." }
```

**好处：** 结构化的点踩原因让管理员能快速分类问题（回答质量问题 vs 检索问题 vs 知识库问题），而不需要逐条阅读完整对话。

#### C2. 后端增强 — 审核后的自动化动作

`AiChatServiceImpl.submitReview()` 之后增加触发器：

```
submitReview(chosenAction) 完成后：

  if (chosenAction == "REVIEWED"):  // 确认问题真实存在
    ├── 1. 如果 feedbackContent 包含 "引用的法条有误"
    │      → 将此条记录加入 Golden Dataset（负面用例：验证模型不再犯同样错误）
    │
    ├── 2. 如果 feedbackContent 包含 "回答不准确" 或 "没有回答我的问题"
    │      → 触发知识库缺口检测（用此问题重新检索 → LLM 判断检索结果是否充足）
    │      → 如果知识库不足 → 加入知识补充待办清单
    │
    └── 3. 记录到 review_log 表（追踪每条反馈的处理结果和时效）
  
  if (chosenAction == "DISMISSED"):
    └── 标记为误触/恶意反馈，不计入质量统计
```

#### C3. 反馈聚合分析

在 `RagMetricsServiceImpl` 中新增 `feedbackBreakdown` 维度：

```
新增 Redis Hash 字段（按天聚合）:
  ├── feedback:reason:inaccurate      # 回答不准确的次数
  ├── feedback:reason:wrong_citation  # 法条引用错误的次数
  ├── feedback:reason:irrelevant      # 答非所问的次数
  ├── feedback:reason:too_vague       # 回答太笼统的次数
  ├── feedback:reason:other           # 其他原因的计数
  ├── feedback:source:hot_cache       # 点踩回答中来自热点缓存的占比
  ├── feedback:source:similar_question # 来自相似问题的占比
  ├── feedback:source:law_knowledge   # 来自知识检索的占比
  └── feedback:source:llm_direct      # 来自 LLM 直接回答的占比
```

**价值：** "来自 LLM 直接回答的点踩率是来自知识检索的 3 倍" → 说明知识库覆盖不足，优先扩充知识库。

#### D2. 持久化改造

**当前问题：** 指标仅存 Redis，14 天过期，无历史数据。

**方案：** 增加 MySQL 持久化层，但不改变当前 Redis 实时查询路径。

```
新增表: rag_metrics_daily
  id, date, total_requests,
  cache_hits, similar_hits, knowledge_hits, llm_direct_count, non_legal_count,
  avg_latency_ms, p50_latency_ms, p95_latency_ms,
  total_likes, total_dislikes,
  avg_top_score, llm_fallback_rate,
  created_at, updated_at

定时任务（每天凌晨 1:00）:
  读取前一天 Redis metrics → 写入 rag_metrics_daily
  → 保留 Redis 作为实时查询层（14 天 TTL 不变）
  → MySQL 作为历史存储层（长期保留，用于趋势分析）
```

#### D3. 简单管理后台页面

不需要完整的前端仪表盘框架。在现有管理后台基础上新增 "质量看板" 页面：

```
页面布局:
┌──────────────────────────────────────────────────┐
│  今日概览                                          │
│  ┌─────────┬─────────┬─────────┬─────────┐       │
│  │ 总请求   │ 点赞率   │ 缓存命中 │ LLM兜底 │       │
```

### Phase E：CI + 自动化回归（优先级 P2，预计 1 天）

#### E1. GitHub Actions 配置

```yaml
# .github/workflows/rag-evaluation.yml
name: RAG Quality Evaluation

on:
  pull_request:
    paths:
      - 'src/main/java/com/lhs/lawmind/service/impl/RagServiceImpl.java'
      - 'src/main/java/com/lhs/lawmind/config/RagConfig.java'
      - 'src/main/resources/application.yml'
  push:
    branches: [main]

jobs:
  evaluate:
    runs-on: ubuntu-latest
    services:
      redis:
        image: redis/redis-stack:latest
        ports: [6379:6379]
      mysql:
        image: mysql:8.0
        env: {...}
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-java@v4
        with: { java-version: 17 }
      - run: mvn test -P evaluation
      - name: Compare with baseline
        run: |
          python scripts/compare_eval_results.py \
            --current target/golden-eval-report.json \
            --baseline docs/eval-baseline.json \
            --threshold 0.05
```

#### E2. 基线对比脚本

```
scripts/compare_eval_results.py:
  读取当前评估报告 + 基线报告
  对比每个指标:
    ├── 如果 Faithfulness 下降 > 5% → CI 失败，阻止合并
    ├── 如果 Answer Relevance 下降 > 5% → CI 警告
    ├── 如果 Source Match 下降 > 0 → CI 失败（硬性退化）
    └── 如果 Keyword Recall 下降 > 10% → CI 警告
  输出 Markdown 格式的对比报告，贴到 PR comment
```

### 推荐执行顺序

| 周次 | 任务 | 理由 |
|------|------|------|
| 第 1 周前半 | Phase A（Golden Dataset 接入） | 快速见效，让已有的 40 条数据发挥作用 |
| 第 1 周后半 | Phase C（反馈闭环） | 用户感知最强，点踩后能采集原因 |
| 第 2 周前半 | Phase B（RAGAS 自建评估） | 为 CI 门禁提供质量信号 |
| 第 2 周后半 | Phase D（在线指标 + 持久化） | 打通历史趋势分析 |
| 第 3 周 | Phase E（CI + 自动化回归） | 把所有评估串联成自动化流程 |

## 五、不纳入本次优化的内容

| 内容 | 理由 |
|------|------|
| A/B 测试框架 | 路线图定为 Phase 5（P2），当前用户量和请求量不足以支撑统计显著性。建议 DAU > 1000 后再做。 |

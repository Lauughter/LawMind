## 一、现状分析

### 1.1 已有的基础设施

| 组件 | 位置 | 状态 |
|------|------|------|
| 请求追踪 ID | `LogTraceInterceptor.java:46` | 已生成 `reqId` 放入 MDC，但 `userId`、`convId` 仅读不写 |
| MDC 异步传递 | `ThreadPoolConfig.java:35-47` | `taskExecutor` 已实现 TaskDecorator，`vectorizeTaskExecutor` 未实现 |
| AOP 日志 | `ControllerLogAspect` / `ServiceLogAspect` / `SchedulerLogAspect` | 三切面覆盖，较完善 |
| 业务指标收集 | `RagMetricsServiceImpl.java` | 基于 Redis Hash + ZSet，14天 TTL，支持 P50/P95 |
| 指标持久化 | `MetricsPersistenceScheduler.java` | 每日凌晨 1:07 cron，Redis → MySQL `rag_metrics_daily` 表 |
| 指标 API | `RagMetricsController.java` | `/admin/metrics/today`、`/trend`、`/quality/*` 端点 |
| Actuator | `application.yml:143-150` | 仅暴露 `health`、`info` |
| Token 成本估算 | `RagPersistenceService.java:234-238` | 硬编码价格常量，仅存于 AiChat 表 |

### 1.2 缺失的关键能力

1. **无 Micrometer 指标注册** — 指标存 Redis，不是 Prometheus 可消费的格式
2. **无分布式链路追踪** — 单次 RAG 调用内部的 6 个子步骤无法串联可视化
3. **无 Grafana 仪表盘** — 运维需要看代码日志或手动调用 `/admin/metrics` API
4. **无告警规则** — 指标异常无人知晓
5. **异步 MDC 不完整** — `vectorizeTaskExecutor` 丢失 `reqId`，RequestContext ThreadLocal 不跨线程传播
6. **日志配置不可定制** — 无 `logback-spring.xml`，格式不可调
7. **MyBatis SQL 日志走 stdout** — 不受日志框架管控，可能丢失

## 四、Phase 1：基础加固（第1周）

> 目标：日志不丢 traceId，异步上下文完整，日志格式可控

### 4.1 LogTraceInterceptor — 写入 userId 和 convId 到 MDC

**当前问题：** `MDC_KEY_USER_ID` 和 `MDC_KEY_CONV_ID` 常量已定义但从未写入 MDC，导致日志模式的 `%X{userId}`、`%X{convId}` 永远为空。

**文件：** `LogTraceInterceptor.java`

**改动：** 在 `preHandle` 中仅写入 `reqId`。`userId` 在 `JwtInterceptor` 认证后写入 `RequestContext`，但此时 `LogTraceInterceptor` 已执行完毕。解决方案：将 `userId` 和 `convId` 的 MDC 写入移到实际产生这些值的时机：

- **userId**：在 `JwtInterceptor.preHandle()` 第 65 行（设置 `RequestContext.setUserId(userId)` 之后）调用 `MDC.put("userId", String.valueOf(userId))`
- **convId**：在 `AiChatController.chat()` / `AiChatController.chatStream()` 中解析出 `conversationId` 之后调用 `MDC.put("convId", conversationId.toString())`

同时在 `LogTraceInterceptor.afterCompletion()` 中保持 `MDC.clear()` 不变（统一清理）。

**JwtInterceptor 改动位置：**
```java
// JwtInterceptor.java — 在 RequestContext.setUserId(userId) 之后增加：
MDC.put("userId", String.valueOf(userId));
```

**AiChatController 改动位置：**
```java
// AiChatController.java — 在解析出 conversationId 之后增加：
if (conversationId != null) {
    MDC.put("convId", conversationId.toString());
}
```

### 4.2 异步 MDC 全量补全 — vectorizeTaskExecutor

**当前问题：** `AsyncConfig.vectorizeTaskExecutor()` 没有 TaskDecorator，向量化异步任务的日志丢失 `reqId`。

**文件：** `AsyncConfig.java`

**改动：**
```java
@Bean(name = "vectorizeTaskExecutor")
public Executor vectorizeTaskExecutor() {
    ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
    executor.setCorePoolSize(5);
    executor.setMaxPoolSize(10);
    executor.setQueueCapacity(25);
    executor.setThreadNamePrefix("Vectorize-");

    // 新增：MDC 传递
    executor.setTaskDecorator(runnable -> {
        Map<String, String> contextMap = MDC.getCopyOfContextMap();
        return () -> {
            try {
                if (contextMap != null) {
                    MDC.setContextMap(contextMap);
                }
                runnable.run();
            } finally {
                MDC.clear();
            }
        };
    });

    executor.initialize();
    return executor;
}
```

### 4.3 RequestContext 跨线程传播

**当前问题：** `RequestContext` 基于纯 `ThreadLocal`，`@Async` 方法中 `RequestContext.getUserId()` 返回 `null`。

**文件：** `ThreadPoolConfig.java` `taskExecutor()` 的 TaskDecorator

**改动：** 在已有 MDC 传播的基础上，增加 RequestContext 快照的传播：
```java
executor.setTaskDecorator(runnable -> {
    Map<String, String> contextMap = MDC.getCopyOfContextMap();
    // 新增：RequestContext 快照
    RequestContext.Snapshot snapshot = RequestContext.getSnapshot();
    return () -> {
        try {
            if (contextMap != null) {
                MDC.setContextMap(contextMap);
            }
            // 新增：恢复 RequestContext
            if (snapshot != null) {
                RequestContext.restore(snapshot);
            }
            runnable.run();
        } finally {
            MDC.clear();
            RequestContext.clear();
        }
    };
});
```

`RequestContext` 需新增两个静态方法：
```java
// 获取快照（调用线程执行）
public static Snapshot getSnapshot() {
    return new Snapshot(userIdThreadLocal.get(), requestIdThreadLocal.get(), 
                        conversationIdThreadLocal.get(), new HashMap<>(extraThreadLocal.get()));
}

// 恢复快照（异步线程执行）
public static void restore(Snapshot snapshot) {
    userIdThreadLocal.set(snapshot.userId);
    requestIdThreadLocal.set(snapshot.requestId);
    conversationIdThreadLocal.set(snapshot.conversationId);
    extraThreadLocal.set(new HashMap<>(snapshot.extra));
}

public record Snapshot(Long userId, String requestId, String conversationId, Map<String, Object> extra) {}
```

同样的 TaskDecorator 也要应用到 `AsyncConfig.vectorizeTaskExecutor()`。

### 4.4 logback-spring.xml — 统一日志格式

**当前问题：** 无自定义日志配置，无法确保每条日志都携带 `reqId`、`userId`、`convId`。

**新增文件：** `src/main/resources/logback-spring.xml`

**格式设计：**
```xml
<?xml version="1.0" encoding="UTF-8"?>
<configuration>
    <!-- 控制台输出：开发环境带颜色 + 完整上下文 -->
    <appender name="CONSOLE" class="ch.qos.logback.core.ConsoleAppender">
        <encoder>
            <pattern>%d{HH:mm:ss.SSS} %highlight(%-5level) [%thread] %cyan(%logger{36}) 
                     | reqId=%-8X{reqId} userId=%-6X{userId} convId=%-6X{convId} 
                     | %msg%n</pattern>
        </encoder>
    </appender>

    <!-- 文件输出：生产环境 JSON 格式（方便日志平台采集） -->
    <springProfile name="prod">
        <appender name="FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
            <file>logs/lawmind.log</file>
            <rollingPolicy class="ch.qos.logback.core.rolling.TimeBasedRollingPolicy">
                <fileNamePattern>logs/lawmind.%d{yyyy-MM-dd}.log</fileNamePattern>
                <maxHistory>30</maxHistory>
                <totalSizeCap>5GB</totalSizeCap>
            </rollingPolicy>
            <encoder>
                <pattern>{"time":"%d{ISO8601}","level":"%level","thread":"%thread",
                         "logger":"%logger{36}","reqId":"%X{reqId}","userId":"%X{userId}",
                         "convId":"%X{convId}","message":"%msg"}%n</pattern>
            </encoder>
        </appender>

        <root level="INFO">
            <appender-ref ref="FILE"/>
        </root>
    </springProfile>

    <springProfile name="dev">
        <root level="INFO">
            <appender-ref ref="CONSOLE"/>
        </root>
        <logger name="com.lhs.lawmind" level="DEBUG"/>
    </springProfile>
</configuration>
```

### 4.5 MyBatis SQL 日志走 SLF4J

**当前问题：** `application.yml` 第 39 行配置 `org.apache.ibatis.logging.stdout.StdOutImpl`，SQL 日志直接输出到 stdout，不走 SLF4J + MDC。

**文件：** `application.yml`

**改动：** 删除 MyBatis 配置中的 `log-impl: org.apache.ibatis.logging.stdout.StdOutImpl`，MyBatis 会自动适配 SLF4J，在 `application-dev.yml` 中增加：
```yaml
logging:
  level:
    com.lhs.lawmind.mapper: DEBUG   # SQL 日志走 SLF4J + MDC
```

## 五、Phase 2：追踪与仪表盘（第2-3周）

> 目标：单次 RAG 调用端到端可追踪，核心指标有 Grafana 面板

### 5.1 接入 Micrometer + Prometheus

**原理：** Spring Boot Actuator 内置 Micrometer，只需添加 Prometheus 注册中心依赖，即可自动暴露 `/actuator/prometheus` 端点供 Prometheus 抓取。

**pom.xml 新增依赖：**
```xml
<dependency>
    <groupId>io.micrometer</groupId>
    <artifactId>micrometer-registry-prometheus</artifactId>
</dependency>
```

**application.yml 改动：**
```yaml
management:
  endpoints:
    web:
      exposure:
        include: health,info,metrics,prometheus,loggers
  metrics:
    export:
      prometheus:
        enabled: true
    tags:
      application: LawMind
  endpoint:
    health:
      show-details: always
```

**验证：** 启动应用后访问 `http://localhost:8080/api/actuator/prometheus`，应看到 JVM 指标（`jvm_memory_used_bytes`、`jvm_gc_pause_seconds` 等）和 HTTP 请求指标。

### 5.2 自定义业务指标 — 用 Micrometer 替代 Redis 指标

**原则：** 保留 Redis 指标收集作为冗余（高可用），但新增 Micrometer 指标作为 Prometheus → Grafana 的数据源。两层并行运行，逐步切换。

**新增类：** `RagMetricsMicrometer.java`（放在 `service/impl/`）

**需要注册的 Meter 指标：**

| 指标名 | 类型 | 标签 | 说明 |
|--------|------|------|------|
| `rag_requests_total` | Counter | source, intent, lawType | 总请求数，按来源/意图/法律类型拆分 |
| `rag_latency_seconds` | Timer | source, phase | 各阶段耗时（pre/embed/search/gen/total） |
| `rag_retrieved_docs` | DistributionSummary | source | 每次检索返回的文档数 |
| `rag_top_score` | Gauge | — | 当前检索最高相似度 |
| `rag_cache_hit_ratio` | Gauge | — | 缓存命中率（1min 滑动窗口） |
| `rag_feedback_total` | Counter | type (up/down), reason | 用户反馈计数 |
| `rag_llm_fallback_rate` | Gauge | — | LLM 兜底比例（知识库无结果的请求占比） |

**使用示例（在 RagServiceImpl 中）：**
```java
private final MeterRegistry meterRegistry;

// 初始化
Counter requestCounter = Counter.builder("rag_requests_total")
    .description("Total RAG requests")
    .tag("source", source)
    .register(meterRegistry);
requestCounter.increment();

// 延迟
Timer latencyTimer = Timer.builder("rag_latency_seconds")
    .description("RAG phase latency")
    .tag("phase", "search")
    .register(meterRegistry);
latencyTimer.record(searchMs, java.util.concurrent.TimeUnit.MILLISECONDS);
```

### 5.3 接入 Langfuse 分布式追踪

**LangChain4j 0.36.0 对 Langfuse 的支持：** LangChain4j 从 0.33.0 开始内置了 Langfuse 集成。可通过 `langchain4j-langfuse` 模块接入。

**pom.xml 新增依赖：**
```xml
<!-- 注意：检查 langchain4j 0.36.0 是否有 langfuse 模块，若无则使用 OpenTelemetry -->
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-langfuse</artifactId>
    <version>0.36.0</version>
</dependency>
```

**若 LangChain4j 0.36.0 不支持 langfuse，替代方案：手动集成 Langfuse SDK**

在 `RagServiceImpl.processQuestion()` 中手动创建 Trace：

```java
// 手动 Langfuse 追踪（使用 Langfuse Java SDK）
import com.langfuse.client.Langfuse;

private final Langfuse langfuse;

public AIChatResponse processQuestion(Long userId, String question, Long conversationId) {
    var trace = langfuse.newTrace()
        .name("rag-query")
        .userId(String.valueOf(userId))
        .input(question)
        .build();

    // Step 1: 预处理
    var preSpan = trace.newSpan().name("preprocess").build();
    // ... 预处理逻辑 ...
    preSpan.end();

    // Step 2: 向量化
    var embedSpan = trace.newSpan().name("embedding").build();
    // ... 向量化逻辑 ...
    embedSpan.end();

    // Step 3: 混合搜索 + RRF
    var searchSpan = trace.newSpan().name("hybrid-search").build();
    searchSpan.setMetadata(Map.of("topK", topK, "hybrid", hybridEnabled));
    // ... 搜索逻辑 ...
    searchSpan.setOutput(retrievedCount + " docs");
    searchSpan.end();

    // Step 4: Rerank
    if (rerankEnabled) {
        var rerankSpan = trace.newSpan().name("rerank").build();
        // ... rerank ...
        rerankSpan.end();
    }

    // Step 5: LLM 生成
    var genSpan = trace.newSpan().name("llm-generation").build();
    genSpan.setMetadata(Map.of("model", modelName, "promptLen", promptLen));
    // ... 生成逻辑 ...
    genSpan.setOutput("done, " + outputTokens + " tokens");
    genSpan.end();

    trace.end();
}
```

## 六、Phase 3：高级可观测性（第4-5周）

### 6.1 告警规则

**告警配置位置：** Prometheus AlertManager 或 Grafana Alerting

| 告警名称 | 触发条件 | 严重级别 | 通知方式 |
|---------|---------|---------|---------|
| LLM 兜底率突增 | `rag_llm_fallback_rate > 0.4` 持续 5min | **严重** | 钉钉/飞书/电话 |
| P99 延迟飙升 | `histogram_quantile(0.99, rate(rag_latency_seconds_bucket[5m])) > 15` | **警告** | 钉钉/飞书 |
| 缓存命中率骤降 | `rag_cache_hit_ratio < 0.2` 持续 10min | **警告** | 钉钉/飞书 |
| 点赞率持续下降 | `rate(rag_feedback_total{type="up"}[1h]) / rate(rag_feedback_total[1h]) < 0.5` 持续 1h | **警告** | 钉钉 |
| Rerank API 失败率 | `rate(rag_rerank_errors_total[5m]) > 0.1` | **严重** | 钉钉/飞书 |
| Redis 连接异常 | `redis_connected_clients < 1` | **严重** | 电话 |
| JVM 堆内存 > 85% | `jvm_memory_used_bytes{area="heap"} / jvm_memory_max_bytes{area="heap"} > 0.85` | **警告** | 钉钉 |

### 6.2 Token 成本追踪增强

**当前：** `RagPersistenceService.calculateCost()` 硬编码价格。**改进：**

1. 将单位 Token 价格写入 `sys_config` 表，支持热更新
2. 新增 Micrometer Counter：`rag_cost_total`（按 model 拆分）
3. Grafana 面板展示每日/每周/每月费用趋势

```java
// 在 LLM 调用完成后
double cost = (inputTokens / 1000.0 * inputPricePer1K) 
            + (outputTokens / 1000.0 * outputPricePer1K);
Counter costCounter = Counter.builder("rag_cost_total")
    .description("Total RAG cost in USD")
    .tag("model", modelName)
    .register(meterRegistry);
costCounter.increment(cost);
```

### 6.3 日志归档与保留策略

| 策略项 | 当前 | 建议 |
|--------|------|------|
| 日志位置 | 默认 stdout | prod: `logs/lawmind.log`，滚动归档 30 天 |
| 日志级别 | dev=DEBUG, prod=INFO | 保持不变 |
| 敏感信息 | 无显式脱敏 | PII 脱敏通过 `PiiUtil.mask()` 在日志输出前处理 |
| SQL 日志 | stdout 直接输出 | 走 SLF4J，dev=TRACE，prod=OFF |

## 七、验收标准

### Phase 1 验收标准
- [ ] 日志中每条记录都包含 `reqId`、`userId`、`convId`（如果有）
- [ ] `@Async("vectorizeTaskExecutor")` 方法中日志不丢失 `reqId`
- [ ] `@Async` 方法中 `RequestContext.getUserId()` 可正常获取
- [ ] 生产日志输出为 JSON 格式，可被日志平台（ELK/Loki）采集
- [ ] MyBatis SQL 日志走 SLF4J，包含 `reqId`

### Phase 2 验收标准
- [ ] `/actuator/prometheus` 端点可访问，返回 JVM + 自定义业务指标
- [ ] Grafana 上可见两个仪表盘（RAG 业务 + 系统资源）
- [ ] Grafana 面板核心指标均有数据且趋势合理
- [ ] Langfuse 上可查到单次 RAG 请求的完整调用链
- [ ] 每次 RAG 请求在 Langfuse 中可以看到各子步骤耗时

### Phase 3 验收标准
- [ ] 6 条告警规则已在 Prometheus AlertManager 或 Grafana Alerting 配置
- [ ] 告警触发后 5 分钟内收到钉钉/飞书通知
- [ ] 管理员可以在 Grafana 上看到每日 Token 费用趋势
- [ ] 生产日志按天滚动归档，保留 30 天

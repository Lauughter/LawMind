## 一、项目现状总览

### 1.1 当前技术栈

| 层级 | 当前选型 | 评价 |
|------|---------|------|
| 后端框架 | Spring Boot 3.5.12 + Java 17 | 合理，可沿用 |
| ORM | MyBatis 3.0.5 | 可沿用 |
| 数据库 | MySQL 8.x | 可沿用 |
| 向量存储 | Redis Stack（Redis Search 向量索引） | 小规模可用，大规模需升级 |
| AI 框架 | LangChain4j 0.36.0 + 阿里云百炼 DashScope | 可沿用 |
| 限流降级 | Sentinel 1.8.6 | 可沿用 |
| 前端 | Vue 3 + Vite 5 + Element Plus | 可沿用 |

### 1.2 当前 RAG 管道

```
用户问题 → 预处理(MD5) → 热点缓存(Redis) → 向量化 → 相似问题检索
→ 法律知识检索 → LLM 生成 → 异步后处理（日志/热点升级/自动学习）
```

**管道特点：**
- 单路向量检索（余弦相似度）
- 简单阈值过滤（≥0.85 相似问题命中，≥0.70 知识库命中）
- 关键词类型匹配辅助过滤
- 支持多轮对话历史上下文
- 支持 SSE 流式输出

## 二、工程架构差距

### 2.1 数据接入层（Ingestion Pipeline）

#### 当前状态
- 单文件上传 → 存储到 `law_file_upload` 表
- 使用 Apache POI 提取 Office 文件文本
- 手动触发向量化，定时任务每 60s 处理 10 条
- 无分块策略，整段内容直接向量化

#### 企业级标准

```
文件上传 → 格式检测 → 文档解析 → 清洗 → 智能分块 → 元数据提取 → 向量化 → 入库
                                                    ↑
                                              失败重试 + 死信队列
```

#### 具体差距与改进方案

| 差距项 | 现状 | 改进方案 | 优先级 |
|--------|------|---------|--------|
| **多格式解析** | 仅支持 Office（POI） | 引入 Apache PDFBox（PDF）、Apache Tika（统一解析）、OCR（Tesseract/阿里云OCR） | P1 |
| **智能分块** | 无分块，整段向量化 | 实现多种分块策略：固定大小分块（512 tokens）、语义分块（按段落/条款）、递归分块。合同类文档按"第X条"拆分效果最佳 | P0 |
| **分块重叠** | 无 | 相邻分块保留 10-20% 重叠，避免语义断裂 | P1 |
| **元数据提取** | 仅存储文件名 | 提取：文档类型（法律/合同/判例）、发布日期、发布机构、效力级别、地域、标签 | P1 |
| **ETL 流水线** | 定时 10 条/批，无重试 | 实现任务队列 + 失败重试（指数退避）+ 死信队列 + 进度追踪 | P2 |
| **批量导入** | 无 | 支持从国家法律法规数据库批量导入结构化法条 | P2 |

#### 建议的 Chunking 架构

```
LawMind 文档分块策略：

法律条文类 → 按"第X条"或"第X章"拆分
合同文本类 → 按条款编号拆分
裁判文书类 → 按"本院认为""判决如下"等段落拆分
通用文本类 → 512 token 固定窗口 + 50 token 重叠
```

### 2.2 检索策略（Retrieval）

> **这是与企业级差距最大的模块，也是优化收益最高的模块。**

#### 当前状态

```
用户问题 → Embedding → Redis 向量搜索(余弦相似度, Top-5) → 阈值过滤 → 喂给 LLM
```

#### 企业级多层检索架构

```
用户问题
  ├── Step 1: 查询理解
  │   ├── 意图分类（咨询/起草/查法条/审合同）
  │   ├── 实体抽取（法律类型、当事人、金额、日期）
  │   └── 查询改写（口语 → 法律专业表述）
  │
  ├── Step 2: 混合检索（多路召回）
  │   ├── 向量检索（语义匹配）
  │   ├── BM25 关键词检索（精确匹配）
  │   └── 元数据过滤（法律类型、时效、地域）
  │
  ├── Step 3: 重排序（Reranking）
  │   ├── 粗排：RRF（Reciprocal Rank Fusion）融合多路结果
  │   └── 精排：Cross-Encoder 模型重新打分（如 bge-reranker-v2-m3）
  │
  └── Step 4: 后处理
      ├── 去重
      ├── 多样性控制（MMR）
      └── 上下文压缩（只保留最相关的片段）
```

#### 具体差距与改进方案

| 差距项 | 现状 | 改进方案 | ROI |
|--------|------|---------|-----|
| **查询改写（Query Rewriting）** | 无，原始问题直接向量化 | 用 LLM 将口语化问题转换为法律专业表述再检索。例："老板不给工资怎么办" → "用人单位拖欠劳动报酬的法律责任与维权途径" | 极高 |
| **HyDE（假设文档嵌入）** | 无 | LLM 先生成假设性答案 → 用答案的向量去检索（通常比问题向量效果好 10-20%） | 高 |
| **混合检索** | 仅向量检索 | Redis 同时支持向量搜索和全文搜索，再加一路 BM25 召回，RRF 融合 | 极高 |
| **Reranking** | 无，直接用向量距离排序 | 引入 bge-reranker-v2-m3 或 Cohere Rerank API 对召回结果重新打分 | 高 |
| **意图分类** | 无 | 训练/提示分类模型，路由到不同处理链路（简单咨询→缓存、复杂问题→深度检索、文书请求→生成模板） | 中 |
| **元数据过滤** | 无 | 检索时过滤已废止的法律、限定特定法律领域 | 中 |
| **多路召回融合** | 单路 | RRF 算法融合向量路和关键词路的排序结果 | 高 |

#### 建议的 RRF 融合公式

```
RRF_score(doc) = Σ 1 / (k + rank_i(doc))

其中 k=60（平滑参数），rank_i 是文档在第 i 路检索中的排名
```

### 2.3 质量评估体系（Evaluation）

#### 当前状态
- 用户 👍/👎 反馈存入 `ai_chat.feedback` 字段
- 反馈数据**未被自动化使用**，仅存储
- 无自动化评估流程

#### 企业级标准

```
离线评估（发版前）              在线评估（运行中）
├── RAGAS 指标                  ├── 用户点赞率
│   ├── Faithfulness           ├── 复问率（同一用户追问比例）
│   ├── Answer Relevance       ├── 回答阅读时长
│   ├── Context Precision      ├── 复制/分享率
│   └── Context Recall         └── 点踩后的退出率
├── Golden Dataset 回归测试
└── A/B 对比测试
```

#### 具体差距与改进方案

| 差距项 | 改进方案 | 优先级 |
|--------|---------|--------|
| **Golden Dataset** | 人工标注 100-200 条标准法律问答对，每次改动 RAG 参数后跑一遍对比分数 | P0 |
| **RAGAS 评估** | 集成 RAGAS 框架，自动计算 Faithfulness、Context Precision/Recall | P1 |
| **A/B 测试** | 两套 Prompt 或检索参数同时跑，统计点赞率差异 | P2 |
| **反馈闭环** | 用户点踩 → 自动记录到审核队列 → 人工审核 → 修正知识库 / 加入回归测试集 | P1 |
| **在线指标看板** | 实时展示缓存命中率、各来源占比、平均相似度、点赞率趋势 | P1 |

#### RAGAS 指标说明

| 指标 | 含义 | 计算方式 |
|------|------|---------|
| **Faithfulness** | 回答是否完全基于检索到的上下文（不编造） | LLM 判断回答中的每句话是否能从检索文档中推断 |
| **Answer Relevance** | 回答与问题的相关程度 | 基于回答反向生成问题，计算与原问题的相似度 |
| **Context Precision** | 检索到的文档中，相关文档的排名是否靠前 | 相关文档在结果列表中的位置加权 |
| **Context Recall** | 所有相关文档是否都被检索到 | 检索到的相关文档 / 所有相关文档 |

### 2.4 可观测性（Observability）

#### 当前状态
- SLF4J 日志 + Actuator（仅暴露 health/info）
- `LogTraceInterceptor` 有基础 traceId 但未在异步方法中传递
- 无业务指标监控

#### 企业级标准

```
每次 RAG 调用须记录：
├── 请求级：用户ID、问题、时间戳
├── 检索级：每路召回的文档数、相似度分布、耗时
├── 生成级：Prompt 长度、Token 消耗、首 Token 延迟、总耗时
├── 结果级：最终来源（cache/similar/knowledge/llm）、用户反馈
└── 聚合级：QPS、P50/P99 延迟、缓存命中率、LLM 兜底比例
```

#### 具体差距与改进方案

| 差距项 | 改进方案 | 优先级 |
|--------|---------|--------|
| **分布式追踪** | 集成 Langfuse（开源，LangChain4j 原生支持）记录每次 RAG 完整链路 | P1 |
| **业务指标仪表盘** | Grafana 面板：缓存命中率、各源占比趋势、平均检索相似度、Token 消耗趋势 | P1 |
| **告警规则** | LLM 兜底率突增 → 知识库可能有问题；缓存命中率骤降 → Redis 可能故障；P99 延迟飙升 → 限流或扩容 | P2 |
| **异步 MDC 传递** | `@Async` 方法中 MDC 丢失，导致日志中 traceId 断裂。需配置 `TaskDecorator` 传递 MDC 上下文 | P1 |
| **Token 成本追踪** | 记录每次 LLM 调用的 Token 数量和费用，按天/按用户统计 | P2 |

#### Langfuse 集成示例（LangChain4j）

```java
// LangChain4j 0.36+ 原生支持 Langfuse
// application.yml 添加：
langchain4j:
  langfuse:
    host: http://localhost:3000
    public-key: pk-xxx
    secret-key: sk-xxx
```

### 2.5 内容安全（Guardrails）

#### 当前状态
- 关键词匹配判断"是否法律相关"
- XSS 过滤（`SecurityUtil`）
- **无输出内容安全审查**

#### 企业级安全架构

```
用户输入
  ├── 输入护栏（Input Guard）
  │   ├── PII 检测与脱敏（身份证号、手机号、银行卡号）
  │   ├── Prompt 注入检测
  │   ├── 敏感话题识别（国家安全、犯罪方法）
  │   └── 非法律问题友好引导
  │
  └── 输出护栏（Output Guard）
      ├── 回答合规性校验（AI 建议是否合法）
      ├── 引用来源验证（引用的法条是否真实存在、是否现行有效）
      ├── 幻觉检测（回答内容与检索文档是否一致）
      └── 格式安全检查（防止 Markdown XSS、链接注入）
```

#### 具体差距与改进方案

| 差距项 | 改进方案 | 优先级 |
|--------|---------|--------|
| **PII 检测** | 正则 + 规则引擎检测并脱敏身份证号、手机号、银行卡号、地址 | P0 |
| **输出幻觉检测** | 用另一个 LLM 调用或 NLI 模型验证回答是否与检索到的知识一致 | P1 |
| **敏感话题过滤** | 维护敏感词库，涉及国家安全、暴力恐怖、色情等内容直接拒答 | P1 |
| **引用来源验证** | 回答中引用的法条名称在知识库中反向检索验证是否存在 | P2 |
| **合规性声明** | 在每段 AI 回答末尾添加「以上内容仅供参考，不构成法律建议」 | P0 |

### 2.6 Prompt 工程

#### 当前状态
- Prompt 硬编码在 Java 字符串中（`RagServiceImpl.java` ~第1100行）
- 修改 Prompt 需要重新编译部署
- 所有法律问题共用一个 Prompt 模板
- 无 Few-shot 示例

#### 企业级做法

```
Prompt 管理平台
├── 模板存储（数据库/配置文件，支持热更新）
├── 版本控制（每次修改可追溯、可回滚）
├── 变量注入（问题类型、历史上下文、检索结果动态填入）
├── Few-shot 示例库（按问题类型加载不同的示例）
└── A/B 测试（新旧 Prompt 效果对比）
```

#### 具体差距与改进方案

| 差距项 | 改进方案 | 优先级 |
|--------|---------|--------|
| **Prompt 模板化** | Prompt 存储在 `sys_config` 表或独立配置文件，支持运行时热更新 | P1 |
| **按类型选择模板** | 劳动法问题、婚姻法问题、合同问题使用不同的 System Prompt | P1 |
| **Few-shot 示例** | 为每种法律问题类型准备 2-3 个高质量的问答示例，作为 Few-shot 注入 Prompt | P2 |
| **Prompt 效果对比** | 记录每条 Prompt 对应的用户点赞率，自动推荐更优 Prompt | P2 |

#### 建议的 Prompt 模板结构

```
system_prompt_template:
  你是一名专业的{law_domain}法律咨询助手。

  回答规范：
  1. 基于提供的法律知识库内容回答
  2. 引用格式：《法律名称》第XX条
  3. {domain_specific_instruction}

  {few_shot_examples}

  对话历史：
  {history_context}

  相关法律知识：
  {retrieved_knowledge}

  用户问题：{question}
```

### 2.7 知识管理

#### 当前状态
- 法律知识存入 MySQL + Redis 向量
- 自动学习从 LLM 回答中提取新知识（无人工审核自动入库）
- 无知识有效期、无去重、无来源追溯

#### 企业级知识管理

```
知识生命周期管理
├── 入库：人工录入 / 批量导入 / 自动学习（待审核）
├── 审核：自动学习生成的知识 → 人工审核 → 正式入库 / 驳回
├── 维护：定期检查知识时效性、自动标记废止法律
├── 去重：入库前检查与现有知识的语义相似度
├── 版本：知识内容变更后保留历史版本
└── 下线：过期/废止知识自动标记不参与检索
```

#### 具体差距与改进方案

| 差距项 | 改进方案 | 优先级 |
|--------|---------|--------|
| **知识审批流程** | 自动学习产生的知识先进入审核队列，人工审核后入库 | P0 |
| **知识有效期** | `law_knowledge` 表增加 `effective_date`、`expiry_date` 字段，检索时过滤 | P1 |
| **知识去重** | 入库前向量搜索检查相似度 > 0.95 的已有知识，提示可能重复 | P1 |
| **来源追溯** | 每条知识记录出处：哪部法律、哪个司法解释、第几条 | P1 |
| **批量导入** | 对接国家法律法规数据库，批量导入结构化法律条文 | P2 |
| **法规变迁追踪** | 记录法律的历次修订，支持"新旧对照"视图 | P2 |

### 2.8 架构与可靠性

| 差距项 | 现状 | 改进方案 | 优先级 |
|--------|------|---------|--------|
| **配置管理** | 单文件 `application.yml`，敏感信息明文 | 拆分为 `application-{profile}.yml`，敏感信息用环境变量或配置中心 | P0 |
| **Redis 降级** | Redis 不可用时返回 null，静默降级 | 实现明确降级链路：Redis 不可用 → MySQL 全文检索 → 有监控告警 | P1 |
| **字段注入改构造器注入** | 108 处 `@Autowired` 字段注入 | 逐步迁移为构造器注入，字段加 `final` | P1 |
| **异常信息泄露** | `RuntimeException` 的 message 直接返回前端 | 自定义业务异常体系，全局异常处理器统一返回通用错误信息 | P1 |
| **列表接口分页** | 多个 `/list` 接口无分页 | 统一使用 `PageResult` 封装分页结果 | P1 |
| **线程池管理** | `AiChatController` 使用静态 `Executors` | 统一使用 Spring 管理的 `ThreadPoolConfig.taskExecutor` | P2 |
| **前端 TypeScript** | 纯 JavaScript | 逐步迁移到 TypeScript，至少对 API 接口定义类型 | P2 |

## 三、业务功能差距

### 3.1 已实现 vs 未实现

| 功能模块 | 当前状态 | 完整度 |
|---------|---------|--------|
| **智能法律咨询** | RAG 问答 + 多轮对话 + SSE 流式 | 70% |
| **合同管理** | 模板 CRUD、合同 CRUD、版本管理 | 60% |
| **合同审查** | 基础版（上传 → LLM 审查 → 返回风险分析） | 30% |
| **知识库管理** | CRUD + 分页搜索 + 手动/定时向量化 | 50% |
| **文书生成** | 仅有模板占位，功能未实现 | 10% |
| **文件上传** | 上传 + Office 文本提取 | 40% |
| **用户系统** | 注册登录 + JWT 双 Token + 个人信息 | 80% |

### 3.2 未实现的业务功能详情

#### 第一梯队（核心业务，用户感知最强）

**1. 案例检索与类案推荐**

用户描述案情（"我被公司无故辞退，工作3年，没有签合同"），系统：
- 检索相似判例，展示案情摘要、判决结果、赔偿金额
- 按相似度排序，标注案号、法院、判决日期
- 统计类似案件的胜诉率和平均赔偿额

**数据来源：** 中国裁判文书网公开数据，需建立案例库和案例向量索引。

---

**2. 法律文书智能生成**

不是简单模板填充，而是 AI 根据用户描述自动生成：
- 起诉状、答辩状、上诉状、再审申请书
- 仲裁申请书、执行申请书
- 证据清单、质证意见
- 法律意见书

**流程：** 用户描述事实 → AI 提取关键要素（当事人、法律关系、诉求）→ 匹配文书模板 → 生成完整文书 → 用户编辑确认 → 导出 Word/PDF

**3. 智能合同审查增强**

当前基础版只有一个 LLM 调用。增强版：

```
合同上传
├── 逐条风险分析（标注每条条款的风险等级：高/中/低）
├── 缺失必要条款检测（违约金条款、争议解决条款、不可抗力条款等）
├── 与标准合同模板的差异对比
├── 单方有利条款识别（过度倾斜某一方的条款）
├── 合规性检查（是否符合最新法律法规）
└── 生成修改建议 + 修改前后对比
```

## 五、技术选型建议

### 5.1 向量数据库评估

| 方案 | 适用场景 | 推荐度 |
|------|---------|--------|
| **Redis Stack（当前）** | 数据量 < 100万条，单机部署 | 可维持到 Phase 3 |
| **Milvus** | 百万级以上向量，需分布式+混合检索 | Phase 4+ 推荐 |
| **Elasticsearch + 向量插件** | 已用 ES 做全文检索的团队 | 若引入 ES 则推荐 |
| **Qdrant** | 轻量级高性能向量数据库，Rust 实现 | 备选 |

**建议：** 当前阶段维持 Redis Stack，Phase 3 后若知识库超过 50 万条或检索延迟超过 500ms，迁移到 Milvus。

### 5.2 Reranker 模型选型

| 模型 | 语言 | 推荐度 |
|------|------|--------|
| **bge-reranker-v2-m3** | 多语言（含中文） | 首选，BAAI 开源，效果最好 |
| **bge-reranker-large** | 中文优化 | 备选，纯中文场景够用 |
| **Cohere Rerank API** | 多语言 | 不想自部署的首选云服务 |

### 5.3 评估与追踪工具

| 工具 | 用途 | 推荐度 |
|------|------|--------|
| **Langfuse** | 分布式追踪 + 评估 | 首选，开源，LangChain4j 原生支持 |
| **RAGAS** | RAG 质量评估 | 首选，社区标准 |
| **Grafana + Prometheus** | 业务指标仪表盘 | 标配 |

### 5.4 内容安全

| 能力 | 方案 |
|------|------|
| PII 检测 | 正则 + Hutool 的 `DesensitizedUtil` |
| Prompt 注入检测 | 开源方案：guardrails-ai 或自建规则引擎 |
| 敏感词过滤 | Hutool `SensitiveUtil` + 自建法律敏感词库 |
| 幻觉检测 | RAGAS Faithfulness 指标 + NLI 模型 |

## 一、现状分析：当前系统有什么，缺什么

### 1.1 已有的

| 组件 | 位置 | 作用 | 局限 |
|------|------|------|------|
| `AiChat` | MySQL `ai_chat` 表 | 存储每轮 Q&A 原文 | 仅日志，不参与推理 |
| `Conversation` | MySQL `conversation` 表 | 分组管理多轮对话 | 仅时间戳 + 标题，无语义信息 |
| `List<ChatMessage>` | AgentRunner 内 `messages` 变量 | 当前请求的上下文窗口 | 请求结束后销毁 |
| `KnowledgeState` | `ContextCompressor` 内部 | 当前请求中积累的知识原子 | 请求结束后销毁 |
| `ContextCompressor` | 压缩调度器 | 压缩超长上下文 | 仅压缩当前请求，不跨会话 |

### 1.2 缺失的

**跨会话记忆能力**。具体表现为：

- 用户今天咨询了"劳动合同纠纷"，明天再问"上次那个合同怎么样了"，系统完全不知道在说什么
- 用户偏好（如"请援引法条原文"、"用通俗语言解释"）每次都需要重新说明
- 用户反复问过的同类问题（如"试用期被辞退怎么赔偿"），系统不会识别这是已解答过的，重复检索知识库
- 合同审查结果不关联到用户记忆，下次上传同类合同时无法提醒"您上次的劳动合同存在竞业限制范围过宽问题"

## 二、核心概念辨析：记忆 vs 上下文

这是最容易混淆的两个概念。用一个类比说清楚：

```
一个正在读案卷的律师

  上下文 (Context)        = 桌面上正摊开的案卷和法条
  上下文压缩 (Compression) = 把冗长的判例摘要精简为关键要点
  记忆 (Memory)           = 律师大脑中"这个客户我认识，他上次..."

  上下文是"现在在讨论什么"
  记忆是"这个用户是谁、历史背景是什么"
```

### 2.1 五维度对比

| 维度 | 上下文 (Context) | 记忆 (Memory) |
|------|-----------------|---------------|
| 生命周期 | 单次请求内 | 跨会话、跨天、永久 |
| 存储位置 | JVM 堆内存 (`List<ChatMessage>`) | MySQL 表 + Redis 向量索引 |
| 容量限制 | ~6000 token（超阈值即压缩） | 理论上无上限（按用户分区） |
| 内容形态 | 原始消息 + 工具调用结果（完整原文） | 提炼后的事实/偏好/摘要（结构化） |
| 获取方式 | 始终完整加载在整个推理循环中 | 按需检索，注入系统提示词 |
| 更新频率 | 每轮 Agent 推理都在变化 | 会话结束后异步提炼，周期性归并 |
| 可变性 | 每轮追加消息，不可修改历史 | 可更新、归并、衰减、删除 |
| 隐私敏感度 | 中等（随请求结束释放） | 高（持久化，需支持用户查看/删除） |
| 类比 | 程序运行时的栈帧 | 数据库中的用户画像 |

### 2.2 两者如何协作

```
用户发送新问题："上次那个劳动合同的竞业限制，我再问一下..."

  1. MemoryRetriever 从记忆库检索到：
     - "用户在 2026-07-10 审查过一份劳动合同（文件名：张三劳动合同.pdf）"
     - "该合同存在竞业限制范围过宽问题（严重）"
     - "用户偏好：要求每条判断必须引用法条原文"

  2. 记忆注入 System Prompt：
     "[用户背景] 该用户上次于 2026-07-10 审查了'张三劳动合同.pdf'，
      其中竞业限制条款被评定为严重风险，主要问题是竞业范围覆盖全行业且无补偿金。"

  3. Agent 推理时：
     - System Prompt 中已有用户背景 → 无需用户重新解释
     - 上下文 (Context) 中只放本次对话的新消息
     - 知识库检索 + 记忆背景 → 精准回答问题
```

## 三、记忆类型体系（借鉴 Claude Code）

Claude Code 将记忆分为 4 种类型，按语义区分而非按存储形态区分。这一设计比传统的"情节记忆/语义记忆"二分法更贴近实际使用场景：

### 3.1 四种记忆类型

```
┌──────────────────────────────────────────────────────────────┐
│                    记忆类型体系                                │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│  👤 用户记忆 (user)                                            │
│  ┌─────────────────────────────────────────────────────┐     │
│  │ 用户的身份、角色、知识水平、长期偏好                       │     │
│  │ 例："HR经理，法律知识中等"                                │     │
│  │ 例："偏好表格形式呈现对比结果"                             │     │
│  │ 例："经常使用民法典，熟悉合同法章节"                       │     │
│  │ 生命周期：长期（衰减周期 180 天）                         │     │
│  └─────────────────────────────────────────────────────┘     │
│                                                               │
│  💬 反馈记忆 (feedback)                                       │
│  ┌─────────────────────────────────────────────────────┐     │
│  │ 用户对系统的纠正、确认、使用偏好反馈                       │     │
│  │ 例："上次你说试用期最长6个月，但根据法律应该是..."          │     │
│  │ 例："不用再解释n+1了，我已经懂了"                          │     │
│  │ 例："这个方案不对，应该按第36条处理"                       │     │
│  │ 生命周期：中期（衰减周期 90 天）                           │     │
│  └─────────────────────────────────────────────────────┘     │
│                                                               │
│  📋 项目记忆 (project)                                        │
│  ┌─────────────────────────────────────────────────────┐     │
│  │ 用户正在处理的案件、事项、合同审查记录                     │     │
│  │ 例："2026-07-10 审查了张三劳动合同.pdf"                   │     │
│  │ 例："正在处理一起竞业限制纠纷案"                           │     │
│  │ 例："李四借款合同存在利率超标问题待跟进"                   │     │
│  │ 生命周期：较短（衰减周期 30 天）                           │     │
│  └─────────────────────────────────────────────────────┘     │
│                                                               │
│  🔗 参考记忆 (reference)                                      │
│  ┌─────────────────────────────────────────────────────┐     │
│  │ 指向外部资源的指针，而非内容本身                           │     │
│  │ 例："上次检索到的指导案例 (2023)京01民终1234号"           │     │
│  │ 例："用户常用的法规：《劳动合同法》第47条"                │     │
│  │ 例："相关司法解释：《民间借贷司法解释》第26条"             │     │
│  │ 生命周期：中期（衰减周期 60 天）                           │     │
│  └─────────────────────────────────────────────────────┘     │
│                                                               │
└──────────────────────────────────────────────────────────────┘
```

### 3.2 为什么用四类型替代情节/语义二分法

| 对比维度 | 情节/语义二分 | 四类型模型 |
|----------|--------------|-----------|
| 区分依据 | 存储形态（摘要 vs 事实） | 使用场景（谁的信息 / 什么用途） |
| 检索策略 | 统一检索 | 按类型差异化检索权重 |
| 衰减策略 | 统一衰减 | 分类型衰减（项目 30d / 参考 60d / 反馈 90d / 用户 180d） |
| 注入方式 | 统一注入 | 用户记忆常驻 / 项目记忆按需 / 反馈记忆精准匹配 |
| 用户可见性 | 不透明 | 用户可查看、编辑、删除 |
| 代码复杂度 | 2 张表，2 个提取器 | 1 张表（type 字段区分），1 个统一提取器 |

核心洞察：**情节记忆和语义记忆的区别是存储实现细节，而四类型区分的是使用语义。** 使用语义决定了检索策略、衰减周期、注入方式和用户可见性——这些才是影响系统行为的关键因素。

## 四、存储架构：统一表 + Redis 向量索引

### 4.1 总体架构

```
┌─────────────────────────────────────────────────────────────┐
│                      存储层级                                │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  第 0 层：原始数据（已有，不可变）                              │
│  ┌─────────────────────────────────────────────────┐        │
│  │ MySQL: ai_chat (Q&A 原文)                        │        │
│  │ MySQL: conversation (会话标题 + 时间)             │        │
│  │ 用途：审计、回放、用户查看历史记录                     │        │
│  └─────────────────────────────────────────────────┘        │
│                                                              │
│  第 1 层：统一记忆存储 ai_memory（新增，核心）                  │
│  ┌─────────────────────────────────────────────────┐        │
│  │ MySQL: ai_memory (所有记忆类型的统一表)            │        │
│  │   type ENUM: USER / FEEDBACK / PROJECT / REFERENCE│        │
│  │   索引: idx_user_type, idx_user_decay, idx_vector │        │
│  │ Redis: 向量索引（复用现有 EmbeddingUtil）           │        │
│  │ 用途：所有跨会话持久记忆的唯一存储                    │        │
│  └─────────────────────────────────────────────────┘        │
│                                                              │
│  第 2 层：上下文 Context（已有，增强注入）                      │
│  ┌─────────────────────────────────────────────────┐        │
│  │ JVM 堆内存: List<ChatMessage>                     │        │
│  │   - SystemMessage (含注入的记忆背景)               │        │
│  │   - UserMessage (当前问题)                         │        │
│  │   - AiMessage + ToolExecutionResultMessage         │        │
│  │ 用途：当前请求的推理窗口                              │        │
│  └─────────────────────────────────────────────────┘        │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 统一记忆表设计

用一张 `ai_memory` 表替代 v1 的 `ai_memory_episodic` + `ai_memory_semantic` + `ai_user_profile` 三张表：

```sql
CREATE TABLE ai_memory (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id BIGINT NOT NULL,

    -- 四类型区分
    type ENUM('USER', 'FEEDBACK', 'PROJECT', 'REFERENCE') NOT NULL,

    -- 记忆内容
    title VARCHAR(100) COMMENT '简短标题（用于列表展示和一级检索）',
    body TEXT NOT NULL COMMENT '记忆正文（一句话事实或简短摘要）',
    summary VARCHAR(300) COMMENT '摘要（从 body 提炼，用于注入时节省 token）',

    -- 溯源
    origin_session_id BIGINT COMMENT '产生此记忆的会话 ID',
    source_session_ids JSON COMMENT '支撑此记忆的所有会话 ID（用于追溯和更新）',

    -- 质量评分
    confidence DOUBLE DEFAULT 0.5 COMMENT '置信度 0-1（多证据来源则升高）',
    importance DOUBLE DEFAULT 0.5 COMMENT '重要性 0-10（归一化到 0-1，用于检索排序）',
    access_count INT DEFAULT 0 COMMENT '被检索命中次数',

    -- 向量（语义检索）
    embedding JSON COMMENT '1536 维向量（JSON 数组，Redis 中另存一份）',

    -- 生命周期
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    last_accessed_at DATETIME DEFAULT CURRENT_TIMESTAMP,

    -- 索引
    INDEX idx_user_type (user_id, type),
    INDEX idx_user_importance (user_id, importance DESC),
    INDEX idx_user_decay (user_id, last_accessed_at, type),
    INDEX idx_origin_session (origin_session_id)
) COMMENT '统一记忆表——LawMind 跨会话记忆的唯一存储';
```

### 4.3 为什么统一表优于多表

| 维度 | 三表方案 (v1) | 统一表方案 (v2) |
|------|-------------|----------------|
| 表数量 | 3（episodic + semantic + profile） | 1 |
| 跨类型查询 | 需要 UNION 或多次查询 | 单表查询，按 type 过滤 |
| 类型扩展 | ALTER TABLE 新增表 | ALTER TABLE 新增 ENUM 值 |
| 检索复杂度 | 分别检索、分别排序、手动合并 | 统一检索，ORDER BY importance DESC |
| 归并操作 | 跨表归并逻辑复杂 | 同表 UPDATE，置信度叠加 |
| 数据迁移 | 3 张表需独立迁移脚本 | 1 张表统一管理 |
| ORM 映射 | 3 个 Entity | 1 个 Entity + type 字段 |

## 五、记忆生命周期

### 5.1 总览

```
用户发起请求
  │
  ├── [检索] MemoryRetriever.search(userId, currentQuestion)
  │      ├── 一级检索：注入所有记忆的 title + type（~200 token 预算）
  │      ├── 二级检索（Agent 按需调用 retrieveMemory 工具）：获取 body 全文
  │      └── 按类型加权排序（用户 > 反馈 > 项目 > 参考）
  │      ↓
  │   注入 System Prompt: "[用户背景] ... [近期事项] ... [偏好] ... [相关参考] ..."
  │
  ├── [推理] Agent 正常执行（上下文窗口 = 系统提示词 + 当前对话）
  │      └── Agent 可在推理中调用 retrieveMemory(topic) 按需获取详细记忆
  │
  └── [提取] 会话结束后，异步执行 MemoryExtractor.extract()
         ├── LLM 分析完整会话
         ├── 按四类型分类提取新记忆 → MySQL ai_memory
         ├── 检测与已有记忆的冲突 → 标记 CONFLICT 或降低置信度
         ├── 向量化并写入 Redis 索引
         └── 用户纠正行为自动触发 feedback 记忆生成

定时任务（每日凌晨 3:00）：
  └── [归并] MemoryConsolidator.consolidate()
         ├── 合并同类型相似记忆（向量相似度 > merge_threshold）
         ├── 按类型差异化衰减（见 5.4）
         ├── 删除 importance < min_importance 的记忆
         └── 更新 MEMORY.md 式的一级索引汇总
```

### 5.2 阶段 1：检索（两级机制）

**为什么需要两级？** 如果每次请求都注入所有记忆的完整 body，当用户累积 100+ 条记忆时，仅记忆就占 3000+ token，远超过可接受的 800 token 预算。两级检索将"知道有什么"和"知道详细内容"分开。

```
第一级：轻量注入（每次请求自动执行）
  │
  ├── 查询：SELECT id, type, title, importance FROM ai_memory
  │         WHERE user_id = ? ORDER BY importance DESC
  │
  ├── 格式化（约 200 token）：
  │     ## 用户记忆概览
  │     ### 项目 (PROJECT)
  │     - [M12] 2026-07-10 审查了张三劳动合同.pdf（竞业限制风险）
  │     ### 反馈 (FEEDBACK)
  │     - [M5] 用户纠正：试用期赔偿按第47条计算
  │     ### 用户画像 (USER)
  │     - [M1] HR经理，法律知识中等
  │     - [M2] 偏好表格形式呈现
  │     ### 参考 (REFERENCE)
  │     - [M8] 相关案例：(2023)京01民终1234号
  │
  └── 注入 System Prompt，Agent 可见

第二级：按需获取（Agent 调用工具或 MemoryRetriever 判断需要时）
  │
  └── Agent 工具: retrieveMemory(memoryId) → body 全文
      或
      MemoryRetriever 根据当前问题语义相似度检索 Top-3 body 自动注入
```

### 5.3 阶段 2：注入格式

检索到的记忆以结构化格式注入 System Prompt 末尾：

```
## 用户记忆（来自历史交互）

### 用户画像
- HR经理，法律知识中等水平
- 偏好：每条判断要求引用法条原文
- 偏好：喜欢表格形式呈现对比信息

### 近期事项
- 2026-07-10：审查了《张三劳动合同.pdf》，发现竞业限制范围过宽（严重）
- 2026-07-08：咨询了竞业限制补偿金标准

### 历史反馈
- 2026-07-08：用户指出试用期辞退应适用《劳动合同法》第47条而非第87条

### 相关参考
- 《劳动合同法》第23条（竞业限制）、第47条（经济补偿计算）
```

**注入原则**：
- 一级注入（title 列表）始终执行，token 预算 ~200
- 二级注入（body 全文）仅在语义相似度 > threshold 或 Agent 主动调用时执行
- 总注入 token 不超过 `max-total-tokens: 800`

### 5.4 阶段 3：提取

```
完整会话记录
  │
  └── MemoryExtractor (LLM 调用，@Async 异步)
       │
       ├── Step 1: 识别记忆类型
       │     判断本次会话产生了哪些类型的记忆
       │     一次会话可能同时产生多种类型的记忆
       │
       ├── Step 2: 按类型提取
       │     USER: 用户角色、知识水平、长期偏好
       │     FEEDBACK: 用户纠正、确认、否定
       │     PROJECT: 合同审查结果、咨询事项、案件进展
       │     REFERENCE: 引用的法条、案例、外部资源
       │
       ├── Step 3: 冲突检测
       │     与已有记忆做语义相似度比较
       │     - 相似且一致 → 合并，confidence ↑
       │     - 相似但矛盾 → 标记 CONFLICT，保留新记忆，降低旧记忆 confidence
       │     - 全新 → 新增
       │
       └── Step 4: 向量化存储
             EmbeddingUtil.embed(body) → Redis 向量索引
             MySQL 写入 ai_memory 记录
```

### 5.5 阶段 4：归并（定时的差异化衰减）

**分类型衰减策略**（借鉴 Claude Code）：

| 记忆类型 | 衰减周期 | 衰减方式 | 理由 |
|----------|---------|---------|------|
| PROJECT | 30 天 | 超过 30 天未访问，importance *= 0.5 | 案件/事项有明确的时间边界 |
| REFERENCE | 60 天 | 超过 60 天未访问，importance *= 0.7 | 法条/案例引用可能长期有效 |
| FEEDBACK | 90 天 | 超过 90 天未访问，importance *= 0.8 | 用户偏好纠正有持续参考价值 |
| USER | 180 天 | 超过 180 天未访问，importance *= 0.9 | 用户画像相对稳定 |

**归并规则**：

```
MemoryConsolidator.consolidate() 每日凌晨执行：

1. 合并相似记忆：
   - 同 type + 向量相似度 > 0.85 → 合并
   - 合并后 confidence = max(c1, c2) + 0.1 * min(c1, c2)
   - 合并后 source_session_ids = array_merge(s1, s2)

2. 类型差异化衰减：
   - 按上表衰减规则执行
   - 被检索命中（access_count 增长）的记忆跳过本轮衰减

3. 清理：
   - importance < 0.2 → 删除（MySQL + Redis 同步删除）
   - source_session_ids 为空的记忆（无溯源）→ 标记待人工审核

4. 每用户记忆数量上限：
   - 超过 max-per-user (200) → 按 importance 排序，删除最低的
```

## 六、不存储排除规则

Claude Code 的记忆系统有明确的"不存储"规则，避免记忆膨胀和噪音。LawMind 继承这一设计并针对法律场景定制：

### 6.1 永远不进入记忆的内容

| 排除项 | 理由 | 存储位置（如果有） |
|--------|------|-------------------|
| 合同原文 | 阅后即焚原则，用户隐私 | 仅当前请求的内存中 |
| 身份证号、手机号、银行账号 | 个人隐私信息 | 不存储 |
| 代码模式、架构设计 | 可从代码库直接读取 | Git 仓库 |
| 单次问候/闲聊 | 无跨会话价值 | 仅 `ai_chat` 日志 |
| 法律知识库已有的法条原文 | 知识库是权威来源，记忆不应重复 | 知识库 `law_knowledge` |
| 系统内部状态（压缩阈值、工具调用细节） | 与用户无关 | 仅日志 |
| Agent 推理过程的中间步骤 | 每次推理路径不同，无复用价值 | 仅 `ai_chat` 日志 |

### 6.2 提取 Prompt 中的排除指令

在记忆提取 Prompt 中明确指示 LLM：

```
## 不要提取以下内容
1. 合同原文、具体金额、身份证号、手机号、银行账号等隐私信息
2. 法律知识库中已有的法条原文（只需记录法条名称和条款号作为 REFERENCE）
3. 本次会话中 AI 的推理过程和技术细节
4. 用户的问候语、感谢语等无信息量的对话
5. 如果一次会话没有产生任何值得跨会话保留的信息，返回空列表

## 提取原则
- 问自己："这条信息在下一周/下一个月的对话中还会有用吗？"
- 如果答案是否定的，不要提取
- 宁可漏提，不要多提——错误的记忆比没有记忆更糟糕
```

## 七、两级检索机制（详细设计）

### 7.1 设计动机

```
用户记忆数量增长曲线：
  第 1 周：~20 条
  第 1 月：~80 条
  第 3 月：~200 条
  第 6 月：~400 条

如果每次注入所有记忆 body：
  20 条 × 平均 50 字 = 1000 字 ≈ 700 token —— 可接受
  200 条 × 平均 50 字 = 10000 字 ≈ 7000 token —— 远超预算
```

两级检索解决的核心问题：**随着记忆增长，注入成本不线性增长。**

### 7.2 检索流程

```
┌─────────────────────────────────────────────────────────────┐
│  第一级：轻量索引注入（自动，每次请求）                          │
│                                                              │
│  SELECT id, type, title, importance FROM ai_memory           │
│  WHERE user_id = ?                                           │
│  ORDER BY importance DESC                                    │
│  LIMIT 30  -- 最多 30 条标题                                  │
│                                                              │
│  格式化输出（~200 token）：                                    │
│  ## 用户记忆索引                                              │
│  ### 项目 (PROJECT)                                          │
│  - [M12] 2026-07-10 审查了张三劳动合同.pdf                    │
│  - [M9] 李四借款合同存在利率超标问题                           │
│  ### 反馈 (FEEDBACK)                                         │
│  - [M5] 用户纠正：试用期赔偿按第47条计算                        │
│  ### 用户画像 (USER)                                          │
│  - [M1] HR经理，法律知识中等                                   │
│  - [M2] 偏好表格形式呈现                                       │
│  ### 参考 (REFERENCE)                                        │
│  - [M8] 相关案例：(2023)京01民终1234号                         │
│                                                              │
│  💡 Agent 可调用 retrieveMemory(id) 获取详细内容               │
└─────────────────────────────────────────────────────────────┘
           │
           ▼
┌─────────────────────────────────────────────────────────────┐
│  第二级：详细内容注入（按需触发）                                │
│                                                              │
│  触发方式 1：语义自动匹配                                      │
│  - EmbeddingUtil.embed(currentQuestion)                      │
│  - Redis 向量相似度检索 → Top-3（相似度 > 0.7）                │
│  - 自动注入匹配记忆的 body + summary                          │
│                                                              │
│  触发方式 2：Agent 主动调用                                    │
│  - Agent 看到一级索引中的 [M12]，判断与当前问题相关              │
│  - 调用工具: retrieveMemory("M12")                            │
│  - 返回 body 全文注入上下文                                    │
│                                                              │
│  触发方式 3：用户显式引用                                      │
│  - "上次那个合同怎么样了" → 语义匹配到 M12                     │
│  - 自动获取 M12 body 注入                                     │
└─────────────────────────────────────────────────────────────┘
```

### 7.3 Agent 工具定义

```java
// 注册到 Agent 的 Tool 列表
@Tool("获取指定记忆的详细内容，用于了解用户的历史背景")
public String retrieveMemory(
    @P("记忆 ID，从用户记忆索引中获取") String memoryId
) {
    AiMemory memory = memoryStore.findById(Long.parseLong(memoryId));
    if (memory == null) return "记忆不存在";
    memory.setAccessCount(memory.getAccessCount() + 1);
    memory.setLastAccessedAt(LocalDateTime.now());
    memoryStore.save(memory);
    return String.format("[%s] %s\n%s", memory.getType(), memory.getTitle(), memory.getBody());
}
```

## 八、反馈记忆行为闭环

这是 Claude Code 记忆系统中最值得借鉴的设计：**反馈记忆不仅记录用户说了什么，还直接塑造未来的行为。**

### 8.1 闭环流程

```
┌──────────────────────────────────────────────────────────────┐
│                                                               │
│  1. 用户纠正："上次你说试用期最长6个月，但我查了应该是..."        │
│                          │                                    │
│                          ▼                                    │
│  2. 提取为 FEEDBACK 记忆                                       │
│     {                                                         │
│       type: FEEDBACK,                                         │
│       title: "用户纠正试用期上限",                               │
│       body: "用户指出试用期根据《劳动合同法》第19条，              │
│              最长不超过6个月，而非之前回答的3个月"                │
│     }                                                         │
│                          │                                    │
│                          ▼                                    │
│  3. 下次相关话题时自动注入                                      │
│     System Prompt 中出现：                                     │
│     "[历史纠正] 用户曾指出试用期上限为6个月（第19条），            │
│      回答此类问题时请优先引用该法条"                             │
│                          │                                    │
│                          ▼                                    │
│  4. Agent 行为改变                                             │
│     - 回答试用期问题时直接引用《劳动合同法》第19条                │
│     - 避免再次给出错误的 3 个月回答                             │
│     - 涉及劳动法问题时优先检索用户曾纠正过的法条                  │
│                          │                                    │
│                          ▼                                    │
│  5. 反馈闭环完成：用户无需再次纠正同一个错误                      │
│                                                               │
└──────────────────────────────────────────────────────────────┘
```

### 8.2 纠正检测策略

在 MemoryExtractor 中，以下模式触发 FEEDBACK 记忆生成：

| 用户语言模式 | 示例 | 生成动作 |
|-------------|------|---------|
| 显式否定 + 纠正 | "不对，应该是..." | 新增 FEEDBACK，降低被纠正记忆的 confidence |
| 补充信息 | "还有一点你漏了..." | 新增 FEEDBACK，补充到相关 PROJECT 记忆 |
| 偏好声明 | "以后都用表格" | 新增或更新 USER 记忆 |
| 确认肯定 | "对，就是这样" | 提升相关记忆的 confidence |
| 质疑 | "你确定吗？" | 标记相关记忆为待验证 |

### 8.3 冲突处理

当新提取的记忆与已有记忆矛盾时：

```
已有记忆：{ type: FEEDBACK, body: "试用期最长3个月", confidence: 0.7 }

新会话中用户纠正："试用期最长6个月，劳动合同法第19条"

处理逻辑：
1. 语义相似度比较 → 检测到冲突
2. 已有记忆 confidence -= 0.3（降至 0.4）
3. 新记忆 confidence = 0.9（用户显式纠正，高置信度）
4. 已有记忆 source_session_ids 追加当前会话 ID（保留溯源）
5. 下次归并时，低 confidence 记忆可能被清理
```

## 九、记忆提取的 LLM Prompt 设计

### 9.1 统一提取 Prompt

```
你是一位法律 AI 助手的记忆管理分析师。你的任务是从对话中提取可跨会话复用的用户信息。

## 记忆类型

### USER（用户画像）
- 用户的身份、职业、角色
- 用户的法律知识水平
- 用户的长期偏好（如"偏好表格"、"要求法条原文"、"通俗语言"）
- 只提取持久性特征，不提取仅本次有效的临时偏好

### FEEDBACK（用户反馈）
- 用户对 AI 回答的纠正或否定
- 用户补充的新信息
- 用户对某类回答方式的偏好声明
- 用户的确认和肯定（增强已有记忆的置信度）
- 格式：{ 纠正内容, 被纠正的记忆ID(如有), 纠正依据 }

### PROJECT（事项/案件）
- 本次合同审查的对象和核心发现
- 本次法律咨询的核心问题和结论
- 用户正在处理的案件或事项进展
- 格式：{ 标题(20字内), 摘要(200字内), 关键事实(3-5条), 重要性(1-10) }

### REFERENCE（外部参考）
- 用户引用的或 AI 推荐的法条、司法解释
- 有参考价值的指导案例案号
- 用户常用的法律资源
- 只记录引用指针，不记录全文

## 排除规则
以下内容**不要提取**：
1. 合同原文、具体金额、身份证号、手机号、银行账号
2. 法律知识库中已有的法条原文
3. AI 的推理过程和技术细节
4. 问候语、感谢语等无信息量的对话
5. 没有跨会话复用价值的内容

## 评估原则
- 问自己："这条信息在下一周/下一个月的对话中还会有用吗？"
- 如果答案是否定的，不要提取
- 宁可漏提，不要多提

## 冲突检测
检查新提取的记忆是否与已有记忆矛盾：
- 一致 → 标记 MERGE，建议提升已有记忆置信度
- 矛盾 → 标记 CONFLICT，提供新旧对比
- 补充 → 标记 APPEND，关联已有记忆 ID

## 已有记忆
{existing_memory_index}

## 对话记录
{full_conversation}

## 输出格式
返回 JSON：
{
  "new_memories": [
    { "type": "USER|FEEDBACK|PROJECT|REFERENCE", "title": "...", "body": "...", "importance": 0.8 }
  ],
  "updates": [
    { "memory_id": 5, "action": "MERGE|CONFLICT|APPEND", "detail": "..." }
  ],
  "empty_reason": null  // 如果无记忆，说明原因
}
```

### 9.2 重要性评分标准（按类型）

| 类型 | 高重要性 (8-10) | 中重要性 (4-7) | 低重要性 (1-3) |
|------|----------------|---------------|---------------|
| USER | 角色 + 知识水平 | 格式偏好 | 临时偏好 |
| FEEDBACK | 纠正法律错误 | 补充信息 | 一般性反馈 |
| PROJECT | 重大风险发现 | 一般咨询 | 简单信息查询 |
| REFERENCE | 关键案例引用 | 常用法条 | 一次性法条查询 |

## 十三、与已有压缩系统的关系

记忆系统与已有的 `ContextCompressor` / `KnowledgeState` 是**互补**关系，不是替代：

```
请求前:   记忆系统注入背景 → System Prompt
请求中:   上下文窗口 (Context) → 消息列表
          └─ 超量时: ContextCompressor 压缩
          └─ 同时: KnowledgeState 积累知识原子
请求后:   记忆系统提炼新记忆 → MySQL + Redis
定时:     MemoryConsolidator 归并
```

| 系统 | 方向 | 时机 | 存储 |
|------|------|------|------|
| ContextCompressor | 向下压缩 | 请求中（上下文过长时） | JVM 堆 |
| KnowledgeState | 向上积累 | 请求中（每轮工具调用后） | JVM 堆 |
| Memory System | 跨时间提取 | 请求后（异步）+ 请求前（检索注入） | MySQL + Redis |

`KnowledgeState` 是"这次请求学到了什么"，记忆系统是"所有历史请求积累了什么"。两者的数据结构（知识原子）可以统一——`KnowledgeAtom` 的 `ArticleAtom`、`CalcAtom` 可以复用为记忆原子的子类型。

## 十四、隐私与合规

- 所有记忆按 `user_id` 分区隔离，检索和删除均限定单用户范围
- 提供 API：`GET /api/memory/list`（查看）、`DELETE /api/memory/{id}`（单条删除）、`DELETE /api/memory/clear`（全部清除）
- 合同审查的"阅后即焚"原则不变：原始合同文本不进入记忆系统，仅提炼后的摘要进入记忆
- 记忆提取 Prompt 中明确："不提取身份证号、手机号、银行账号等个人隐私信息"
- 记忆溯源：每条记忆记录 `origin_session_id` 和 `source_session_ids`，用户可追溯"系统为什么会记住这个"
- 用户可随时删除记忆，删除操作同步清除 MySQL 记录和 Redis 向量索引

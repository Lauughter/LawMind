# LawMind 质量评估体系文档

## 概述

LawMind 质量评估体系围绕 **Golden Dataset** 构建，覆盖"数据集管理 → 多维评估 → 报告持久化 → 管理看板 → CI 回归门禁 → 反馈闭环"完整链路，确保 RAG 管道每次变更后质量可量化、可追溯、可拦截。

## 一、Golden Dataset（黄金测试集）

### 文件位置

`docs/golden-dataset-rag-evaluation.json`

### 数据结构

```json
{
  "id": "GR001",
  "question": "加班不给加班费怎么办？",
  "intent": "LEGAL_CONSULTATION",
  "expected_law_type": "中华人民共和国劳动法",
  "difficulty": "easy|medium|hard",
  "key_points": ["加班", "加班费", "劳动仲裁"],
  "expected_answer_contains": ["加班费", "劳动法", "支付"],
  "source_requirement": "law_knowledge|hot_cache|similar_question|llm_direct|non_legal_reject",
  "forbidden_content": "暴力,违法方法",
  "min_retrieval_count": 3,
  "review_notes": "人工审核补充说明"
}
```

### 字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | String | 唯一标识，建议格式 `GR` + 时间戳 |
| `question` | String | 测试问题（必填） |
| `intent` | String | 意图分类，如 `LEGAL_CONSULTATION` |
| `expected_law_type` | String | 期望引用的法律类型（可选） |
| `difficulty` | String | 难度：easy / medium / hard |
| `key_points` | List | 问题涉及的关键点 |
| `expected_answer_contains` | List | 期望回答中包含的关键词 |
| `source_requirement` | String | 期望的数据来源路径 |
| `forbidden_content` | String | 回答中不应出现的禁止内容（逗号分隔） |
| `min_retrieval_count` | Integer | 最低检索条数要求 |
| `review_notes` | String | 人工审核备注 |

### 自动增长机制

当管理员审核用户点踩反馈并确认后，系统自动将问题追加到 Golden Dataset：

- **法条引用错误** (`wrong_citation`) → 自动写入 `ADD_TO_GOLDEN_DATASET`
- 新增条目的 `id` 格式为 `GR{timestamp}`

## 二、评估维度（8维度 + 综合得分）

### 2.1 规则评估维度

| 维度 | 方法 | 评分方式 | 权重 |
|------|------|----------|------|
| **来源匹配** `sourceMatch` | 检查回答来源是否符合预期（知识库/缓存/LLM兜底/拒答） | 布尔值 | 计入总分 |
| **关键词召回** `keywordRecall` | 期望关键词在回答中的命中率 | 0.0–1.0 | 计入总分 |
| **法条类型匹配** `lawTypeMatch` | 检索结果中是否包含期望的法律类型 | 布尔值 | 计入总分 |
| **最低回答长度** `answerMinLength` | 回答长度 ≥ 50字符 | 布尔值 | 计入总分 |
| **禁止内容清洁** `forbiddenContentClean` | 回答中不包含禁止关键词 | 布尔值 | 单独统计 |
| **最低检索满足** `minRetrievalOk` | 检索结果条数 ≥ 最低要求 | 布尔值 | 单独统计 |

### 2.2 RAGAS LLM 评估维度

| 维度 | 方法 | 说明 |
|------|------|------|
| **忠实度** `faithfulness` | RAGAS Faithfulness | 回答是否完全基于检索到的上下文，有无编造 |
| **答案相关性** `answerRelevance` | RAGAS AnswerRelevancy | 回答是否直接回应了用户问题 |

### 2.3 综合得分

```
totalScore = (sourceMatch + keywordRecall + lawTypeMatch + answerMinLength) / 4
```

- 综合得分 ≥ 0.5 → **通过** (passed)
- 综合得分 < 0.5 → **失败** (failed)

## 三、核心组件

### 3.1 GoldenDatasetLoader

```
位置: src/main/java/com/lhs/lawmind/evaluation/GoldenDatasetLoader.java
```

从 JSON 文件加载 Golden Dataset 记录列表，默认路径为 `docs/golden-dataset-rag-evaluation.json`。

### 3.2 GoldenDatasetEvaluator

```
位置: src/main/java/com/lhs/lawmind/evaluation/GoldenDatasetEvaluator.java
```

核心评估器，依赖 `RagService`、`RagasEvaluationService`、`EvalReportPersistenceService`。

主要方法：
- `evaluate()` — 使用默认路径执行评估
- `evaluate(String datasetPath)` — 使用指定路径执行评估，自动持久化报告
- `evaluateRecords(List<GoldenDatasetRecord>)` — 对内存中的记录列表评估（供测试使用）

### 3.3 RagasEvaluationService

```
位置: src/main/java/com/lhs/lawmind/evaluation/RagasEvaluationService.java
```

基于 LangChain4j 的 RAGAS 双维度评估服务，调用 LLM 评估忠实度和答案相关性。

### 3.4 EvalReportPersistenceService

```
位置: src/main/java/com/lhs/lawmind/evaluation/EvalReportPersistenceService.java
```

将评估报告写入 MySQL `evaluation_report` 表，包含所有维度平均值和完整 JSON。

## 四、反馈闭环

### 流程

```
用户提问 → 获得回答 → 用户点踩(反馈原因) → 管理员审核
                                                    │
                          ┌─────────────────────────┼──────────────────────────┐
                          ▼                         ▼                          ▼
                   ADD_TO_GOLDEN_DATASET       KNOWLEDGE_GAP            CONFIRMED_ISSUE
                   (法条引用错误)              (回答不准确)              (通用问题确认)
                          │                         │                          │
                          ▼                         ▼                          ▼
                  追加到Golden Dataset        写入review_log             写入review_log
```

### 触发条件

`AiChatServiceImpl.triggerPostReviewActions()` 在管理员审核确认后自动执行：

| 反馈内容匹配 | 动作 |
|-------------|------|
| `wrong_citation` 或 "引用的法条有误" | 追加到 Golden Dataset + 写入 review_log |
| `inaccurate` / `irrelevant` / "回答不准确" / "没有回答我的问题" | 标记 KNOWLEDGE_GAP + 写入 review_log |
| 所有已确认问题 | 写入 review_log (CONFIRMED_ISSUE) |

### 管理员审核 API

| 端点 | 说明 |
|------|------|
| `GET /api/admin/chat/review/list` | 获取待审核列表（分页） |
| `POST /api/admin/chat/review` | 提交审核结果 |

## 六、CI 回归门禁

### 基线对比脚本

```
位置: scripts/compare_eval_results.py
```

**功能：**
- 读取 `target/evaluation/golden-eval-report.json`（评估报告）
- 与 `scripts/eval-baseline.json`（基线）逐维度对比
- 超越容忍阈值则标记为 REGRESSION
- 首次运行自动创建基线

**用法：**
```bash
# 与基线对比（CI 中使用）
python scripts/compare_eval_results.py

# 用当前结果更新基线
python scripts/compare_eval_results.py --update-baseline

# 指定报告路径
python scripts/compare_eval_results.py --report path/to/report.json
```

**退出码：**
- `0` — 通过（无回归或仅改善）
- `1` — 发现回归
- `2` — 参数/文件错误

### 容忍阈值

| 维度 | 阈值 |
|------|------|
| sourceMatch, lawTypeMatch, answerMinLength, passedCases, totalScore | 0.05 |
| keywordRecall, faithfulness, answerRelevance | 0.10 |
| forbiddenContentClean | 0.02 |
| minRetrievalOk | 0.05 |

### CI 工作流

```
位置: .github/workflows/rag-evaluation.yml
```

**触发条件：**
- `workflow_dispatch`（手动触发，支持自定义数据集路径）
- `pull_request` 修改以下文件时自动触发：
  - `RagServiceImpl.java`
  - `RagConfig.java`
  - `application.yml`

**流水线步骤：**
1. 启动 Redis + MySQL 服务容器
2. 执行数据库迁移（V*.sql）
3. 运行评估测试（`mvn test -P evaluation`）
4. **Regression Gate** — 运行基线对比脚本
5. 上传评估报告制品（保留30天）

## 八、日常使用流程

### 1. 查看质量现状

打开管理端 → 质量看板，查看今日请求量、缓存命中率、好评率、LLM兜底率等核心指标。

### 2. 运行评估

```bash
mvn test -P evaluation
```

### 3. 检查回归

```bash
python scripts/compare_eval_results.py
```

如果发现回归，查看对比表确定哪些维度下降，排查对应的代码变更。

### 4. 更新基线

当评估指标有明显改善且确认无误后：

```bash
python scripts/compare_eval_results.py --update-baseline
```

### 5. 处理用户反馈

1. 管理端 → 审核列表，查看点踩记录
2. 审核确认后，系统自动将法条引用错误追加到 Golden Dataset，将知识缺口写入 review_log
3. 定期检查 review_log 中未处理的记录，补充知识库

### 6. 关注趋势

质量看板支持7/14/30天趋势查看，关注 LLM 兜底率是否持续上升（说明知识库覆盖不足）、好评率是否下降等信号。
